## 1. 项目一句话定义

**MarioTrickster** 是一个基于 Unity 2022.3 LTS 开发的 **2D 非对称双人平台跳跃游戏**。一名玩家扮演闯关者 Mario，目标是在关卡中前进并抵达终点；另一名玩家扮演 Trickster，能够伪装成关卡中的障碍、地形或怪物，通过误导、操控和伏击来阻止 Mario 通关。

从项目原始设计意图看，它并不是一个“普通马里奥关卡编辑器项目”，也不是一个“纯美术资产项目”，而是一个围绕 **“非对称平台跳跃 + 伪装对抗 + 可重复关卡博弈”** 展开的完整游戏原型。

| 项目维度 | 当前定义 |
|---|---|
| 游戏类型 | 非对称双人本地对抗 2D 平台跳跃 |
| 核心角色 | Runner（Mario）与 Trickster（伪装者） |
| 核心差异化 | Trickster 可伪装为关卡元素并在游玩中动态干扰 |
| 当前引擎 | Unity 2022.3 LTS（2D Core） |
| 当前开发方式 | 设计驱动，AI 协作，GitHub 持续存档 |
| 当前关键特征 | **关卡白盒工具链已成型，美术生产流程在建立中，正式资产尚未批量接入** |

## 2. 当前最重要的总判断

当前项目**并没有真的“偏离原来的关卡设计主线”**，而是进入了一个很容易让人误以为“换项目了”的阶段：**在关卡白盒工具链已经基本成熟之后，项目开始补“美术生产管线”这条支撑线**。这条支撑线的目标，不是替代关卡设计，而是为已经存在的白盒关卡系统提供后续换肤、风格统一和量产素材能力。

换句话说，项目历史的正确理解应该是：**先验证玩法与关卡逻辑，再准备可持续美术生产，最后回到关卡内容量产和整体验证**。你现在感觉“原来做的是关卡设计，后来却一直在做美术流程”，本质上是因为当前阻塞点恰好落在美术资产供给侧，而不是因为关卡主线被废弃了。

| 结论问题 | 判断 |
|---|---|
| 项目是不是从“关卡设计”变成了“美术项目”？ | **不是** |
| 原来的关卡管线是不是已经失效？ | **不是，仍然存在且可直接继续使用** |
| 当前为什么大量出现美术文档和 LoRA 流程？ | 因为项目进入了 **白盒转可视化** 的准备阶段 |
| 当前真正卡住的地方是什么？ | **美术资产验证与量产路径还没彻底打通** |
| 现在要不要“回到原来关卡管线”？ | **要，而且完全可以马上回去** |

## 3. 项目已经完成到什么程度

从代码与工具层面看，这个项目已经不是一个概念草图，而是一个具备较强可操作性的可玩原型。核心控制、伪装系统、关卡元素、测试工具、ASCII 关卡生成与主题换肤入口都已经建立。真正薄弱的部分主要集中在**视听表现、正式素材、美术风格统一、音效动画和联网**，而不是关卡生成本身。

| 子系统 | 当前状态 | 说明 |
|---|---|---|
| 基础角色控制 | 已完成 | Mario 与 Trickster 的移动、跳跃、受击、基础交互已存在 |
| 伪装与对抗机制 | 已完成 | 包括伪装、扫描、能量、道具操控等核心对抗逻辑 |
| 关卡元素系统 | 已完成 | 地刺、摆锤、火焰、弹跳平台、单向平台、崩塌平台、隐藏通道等均已实现 |
| 游戏流程 | 已完成 | 胜负判定、回合重置、暂停、计时等基础流程闭环已形成 |
| 自动化测试 | 已完成安全网 | 当前有 **135 个自动化测试** 作为防回归底线 |
| 关卡白盒生产 | 已完成主框架 | 已有 ASCII 生成器、模板库、关卡工作台、物理约束与可达性校验 |
| 主题换肤系统 | 已完成框架 | 已可对白盒元素应用 Theme Profile，但缺正式量产素材 |
| 正式美术资产量产 | 未完成 | 当前处于建立风格、验证 LoRA、准备批量生产阶段 |

| 联机 | 未开始 | 仍处于远期阶段 |

这意味着：**项目的玩法骨架和关卡生产骨架已经具备继续设计、继续测试、继续迭代的条件**。如果暂时不追求精美画面，完全可以立即继续做关卡设计、关卡拼装、节奏验证与玩法打磨。

## 4. 当前开发阶段到底在哪里

按照项目路线图，Sprint 1 的核心机制验证已经完成，Sprint 2 的“体验提升与工具链”也基本完成，其代表性成果就是 **Test Console / Level Studio**、ASCII 白盒生成器、主题系统以及围绕关卡设计的工作流文档。按理说，后面应当进入 Sprint 3，也就是**视听打磨与平衡阶段**。

但当前实际推进方式，并不是传统团队里“关卡设计师继续做图、美术同时供图”的并行模式，而是单人/AI 协作模式。因此，一旦美术资产供给没有形成稳定闭环，整个项目就会表现为：**关卡工具已经有了，但因为正式素材还没有统一风格，注意力被迫转移到美术生产流程**。

| 阶段 | 实际状态 | 备注 |
|---|---|---|
| Sprint 1：核心机制验证 | **已完成** | 核心玩法原型已成立 |
| Sprint 2：关卡工具链与体验工具 | **已完成核心闭环** | Level Studio、ASCII 白盒、主题换肤入口已建立 |
| Sprint 3：视听打磨与平衡 | **理论上应进入，实际卡在美术资产供给侧** | 当前主要卡点 |
| Sprint 4：联机与发布 | **未开始** | 远期目标 |

所以，如果让其他 AI 判断项目当前处境，最准确的说法不是“项目改方向了”，而是：

> **项目已经完成玩法和白盒工具验证，当前卡在“如何用稳定、低成本、可复用的方式为现有关卡系统提供统一美术资产”这一步。**

## 5. 为什么会产生“原来在做关卡，现在却跑到美术去了”的错觉

这个错觉非常合理，因为项目文档和最近任务确实大量集中在美术教程蒸馏、提示词配方、概念锚点、LoRA 训练与本地验证上。对外表象看，像是项目从“做关卡”变成了“研究出图”。但从项目结构看，美术管线只是为了服务后续三件事：第一，让白盒关卡能以统一风格呈现；第二，让 Trickster 可伪装对象、敌人、背景和地块可以批量生产；第三，降低后续新增关卡时的素材成本。

真正的问题不在于“有没有美术管线”，而在于**当前美术管线与关卡设计主线之间缺少一份清晰的桥接说明**。用户因此会感到：原来的主线是“做关卡”，现在却一直被要求讨论 LoRA、ComfyUI、训练参数和本地验证，仿佛回不去了。

| 表面现象 | 实际含义 |
|---|---|
| 最近大量文档在讨论美术教程蒸馏 | 在建立长期可复用的出图规则库 |
| 最近任务聚焦 LoRA 训练与验证 | 在为统一风格资产量产做前置准备 |
| README 出现大量“白话美术入口” | 在降低后续跨对话接手的沟通成本 |
| 当前待办最高优先级是 `trickster_style` 验证 | 因为美术供给侧是当前瓶颈 |
| 用户感觉“关卡主线消失了” | 因为**桥接路径没有被明确写出来** |

---

## 6. 关卡管线与美术管线的真实关系

这两个管线不是竞争关系，也不是“旧的被新的替代”。正确关系应该理解为：**关卡管线是主生产骨架，美术管线是附着在其上的视觉供给系统**。

具体来说，关卡管线负责回答“这一关好不好玩、节奏通不通、博弈成立不成立、是否可通关”；美术管线负责回答“这一关长什么样、风格是否统一、资产能否批量生成、是否便于后续换肤和量产”。如果没有关卡主线，美术就会变成无目标出图；如果没有美术支撑，关卡虽然能玩，但难以进入真正的内容生产期。

| 维度 | 关卡管线 | 美术管线 |
|---|---|---|
| 核心目的 | 验证玩法、节奏、可达性、博弈结构 | 提供统一视觉、批量素材与后续换肤能力 |
| 当前成熟度 | **高** | **中低** |
| 当前工具 | Level Studio、ASCII 模板、验证器、片段库、Theme Profile | 提示词配方库、教程蒸馏、概念锚点、LoRA 名录、本地验证模板 |
| 是否必须先完成 | **必须优先成立** | 应服务于前者 |
| 当前阻塞位置 | 主线可继续，但缺统一正式素材 | **当前主要瓶颈** |
| 是否能脱离另一方独立推进 | 可用白盒独立推进 | 不宜脱离关卡目标单独长期推进 |

可以把它理解成这样的顺序：

1. **先用白盒和 ASCII 模板把关卡做出来。**
2. **确认这个关卡在 Mario 与 Trickster 的对抗里好玩。**
3. **再把通过验证的白盒关卡换肤。**
4. **最后把成熟的视觉规则沉淀为可批量生产的资产流程。**

当前项目的问题，不是顺序设计错了，而是**在第二步和第三步之间停得有点久**，导致体感上像“主线转移了”。

## 7. 原来的关卡管线其实还在，而且入口非常明确

从现有工具链来看，关卡设计主线不但还在，而且已经比项目早期更成熟。现在回到关卡管线，并不需要“重建系统”，只需要重新把注意力切回 **Level Studio → ASCII 白盒 → 可玩性验证 → 必要时换肤** 这条路。

当前最关键的操作入口是 **Level Studio（也就是 Test Console）**。它仍然是项目内统一的关卡工作台，支持模板生成、自定义 ASCII 文本构建、片段拼装、单元素投放、主题换肤、传送测试和作弊开关。换句话说，**原来的关卡生产环境没有消失，只是近期没有被拿来当主叙事中心。**

| 入口 | 当前可用性 | 用途 |
|---|---|---|
| `Ctrl+T` / `MarioTrickster → Test Console` | 可用 | 打开统一工作台 |
| Level Builder | 可用 | 生成白盒关卡、拼 ASCII 模板 |
| Custom Template Editor | 可用 | 粘贴/修改 ASCII 文本并一键构建 |
| Element Palette | 可用 | 快速投放单个元素测试机制 |
| Theme Profile + Apply Theme | 可用 | 白盒验证后再换肤 |
| Teleport / Cheats | 可用 | 快速验证关卡局部和对抗节奏 |

这说明，所谓“回到原来的关卡管线”，其实不是回头重做，而是**重新启用已经成熟但最近没有被放在叙事中心的那条主线**。

## 8. 如果现在要回到关卡主线，应该怎么回

最重要的一点是：**不要等美术全搞定再回去做关卡。** 这个项目当前最合理的做法，是把“关卡设计继续推进”和“美术供给侧逐步补齐”重新拆成主次关系。也就是说，关卡继续往前跑，美术作为辅助资源逐步接上，而不是让美术成为整个项目的总闸门。

### 8.1 立即可执行的回归策略

当前最推荐的恢复路径，是把接下来的工作重新划分为三个层次：先恢复白盒关卡设计，再只在必要处使用现有主题换肤，最后把美术 LoRA 验证作为并行支线而不是总阻塞点。

| 层次 | 做什么 | 为什么 |
|---|---|---|
| 第一层：立刻恢复 | 继续用 ASCII 模板和 Level Studio 设计、拼装、测试关卡 | 不依赖正式美术，马上能推进主玩法 |
| 第二层：局部可视化 | 对已经验证通过的白盒关卡，使用 Theme Profile 做最低成本换肤 | 让成果更可看，但不阻塞设计 |
| 第三层：美术并行支线 | 继续验证 `trickster_style`，为未来批量资产做准备 | 保留长期价值，但不再独占主线 |

### 8.2 Unity 内的实际回归动作

如果其他 AI 需要一个非常具体的“怎么回”的操作说明，可以直接照以下流程理解：

1. 打开 Unity 工程。
2. 用 `Ctrl+T` 打开 **Level Studio / Test Console**。
3. 在 **Level Builder** 中优先使用 **Custom Template Editor**，把现有 ASCII 模板或新生成的 ASCII 文本粘进去。
4. 点击 **Build From Text** 直接生成白盒关卡。
5. 用 Teleport 与 Cheats 快速测试局部区域，验证 Mario 路径是否成立、Trickster 是否有足够博弈点。
6. 不要先纠结精美画面，优先修改文本矩阵来调整跳跃、陷阱、分支和节奏。
7. 当某个关卡白盒已经“好玩且可通关”后，再选择是否应用 Theme Profile 做初步换肤。
8. 只有当发现某类元素的正式美术需求反复出现，才回头把它列入美术量产清单。

这个流程意味着：**回到关卡主线并不是“停掉美术”，而是把美术重新放回它应该服务的位置。**

## 9. 当前最真实的阻塞点是什么

当前真正的阻塞，不是“不会做关卡”，也不是“关卡系统失效”，而是**正式美术资产供给尚未形成可持续闭环**。目前仓库里美术管线已经沉淀了大量规则、白话入口、教程蒸馏和 LoRA 名录，但最关键的一步——`MarioTrickster_Style_epoch_10.safetensors` 的本地验证实测结果——仍然缺失。因此，美术方面还无法稳定给出“推荐权重、触发词甜区、污染模式、专属去污词”等真正能指导量产的参数。

这会带来两个后果。第一，继续做纯美术讨论会越来越抽象，因为缺少验证结果。第二，如果因为这个卡点而暂停关卡设计，就会让整个项目节奏失衡。

| 当前阻塞 | 具体表现 | 严重性 |
|---|---|---|
| LoRA 本地验证未完成 | 还没有实测回填推荐权重与污染信息 | 高 |
| 正式素材未批量接入 | 白盒可玩，但正式观感未成型 | 中高 |
| 关卡主线叙事被美术支线遮蔽 | 用户主观感受是“不知道怎么回到关卡” | 高 |
| 关卡与美术之间缺少桥接文档 | 造成决策与优先级混乱 | 高 |

因此，当前最合理的管理结论是：

> **把 LoRA 验证从“总阻塞”降级为“重要并行支线”，同时恢复关卡白盒设计作为主推进轴。**

## 10. 现在最推荐的项目策略

如果目标是让项目重新进入稳定推进状态，我建议采用 **“白盒关卡继续跑，美术验证并行补，换肤只服务已验证内容”** 的策略。这是当前效率和质量都最优的方案。

| 优先级 | 建议动作 | 目标 |
|---|---|---|
| 最高 | 恢复关卡主线，继续拼装和验证 1~2 个完整白盒关卡段 | 让项目重新回到“可玩的内容生产”状态 |
| 高 | 使用现有 8 个 ASCII 模板与片段库做组合，而不是从零发散设计 | 提高节奏、减少空转 |
| 高 | 把每个关卡段的 Trickster 博弈点明确记录为设计目标 | 防止关卡只剩平台跳跃，没有对抗特色 |
| 高 | 继续进行 `trickster_style` 本地验证，但改为并行支线 | 不再让美术成为唯一主任务 |
| 中 | 对已经通过验证的白盒段做最低成本换肤展示 | 帮助观察视觉问题与展示效果 |
| 中低 | 暂时不要推进联网、复杂 UI、大规模系统扩建 | 当前不是主要收益点 |

从产品推进角度说，下一阶段最重要的不是“再蒸馏多少美术教程”，而是**重新产出一批能体现 MarioTrickster 非对称博弈特色的可玩关卡内容**。只要关卡内容继续增长，美术支线的价值就会变得非常明确，因为它将直接服务于已有可玩内容，而不再显得悬空。

## 11. 给其他 AI 的接手结论

如果这份文档要转交给另一个 AI，我建议它先接受以下几个前提判断，否则很容易再次误判项目方向：

| 接手前必须理解的结论 | 说明 |
|---|---|
| 结论 1 | **项目主线仍然是关卡设计与玩法验证，不是纯美术生产。** |
| 结论 2 | **美术管线的存在是为了服务白盒转量产，不是为了取代关卡设计。** |
| 结论 3 | **当前最成熟的可执行能力是关卡白盒工具链，而不是正式资产量产。** |
| 结论 4 | **当前真正卡点是 LoRA 验证尚未闭环，但这不应阻止继续做关卡。** |
| 结论 5 | **“怎么回到原来的关卡管线”的答案就是：直接重新使用 Level Studio 和 ASCII 工作流。** |

另一个 AI 在给出建议时，应该重点回答以下问题，而不是再次把重心全部放到美术学习上：

1. 在现有 8 个模板与片段库基础上，下一批最值得做的关卡组合是什么。
2. 如何让每个关卡段都同时兼顾 Mario 跳跃挑战与 Trickster 伪装/伏击机会。
3. 哪些关卡段值得优先做 Theme Profile 换肤，哪些继续保持白盒即可。
4. LoRA 验证完成后，优先量产哪些类型的资产，才能最大化服务现有关卡系统。
5. 如何把“关卡白盒迭代”与“美术量产”重新排成主次清晰的开发节奏。
