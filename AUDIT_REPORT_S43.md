# S43 审计报告：关卡生成验证系统 — 行业对标与差距分析

> **审计范围**：AsciiLevelValidator.cs、PhysicsMetrics.cs、AsciiLevelGenerator.cs（内置模板）、LevelSnippetLibrary.cs（片段库）、AI_PROMPT_WORKFLOW.md（提示词）、ASCII_Templates.md（模板集）、LevelStudio_DesignGuide.md（设计指南）、PHYSICS_METRICS_GUIDE.md（物理文档）、MarioTrickster_Testing_Guide.md（测试指南）
>
> **审计日期**：2026-04-06（Session 43）

## 一、行业方法论层级（从研究中提炼）

通过研究 Cloudberry Kingdom（GDC/Gamasutra）、MarioGPT（AAAI 2023）、Sturgeon/Metatiles（AIIDE 2020-2022）、VGLC 数据集格式、error454 平台物理方程等行业资料，总结出以下验证方法层级：

| 层级 | 方法 | 可靠性 | 复杂度 | 代表项目 |
|------|------|--------|--------|----------|
| **L1 黄金标准** | Player AI 模拟（让 AI 实际跑通关卡） | 最高 | 高 | Cloudberry Kingdom |
| **L2 图搜索** | A\*/BFS 可达性图分析 | 高 | 中高 | MarioGPT 用 A\* 验证可玩性 |
| **L3 约束求解** | WFC/ASP + 可达性 metatiles | 高 | 中高 | Sturgeon (Cooper 2022) |
| **L4 启发式检查** | 物理约束规则检查（间隙/高度/连通性） | 中 | 低 | **← 我们当前的方案** |
| **L5 无验证** | 仅靠模板/规则生成，不验证 | 低 | 最低 | 简单原型 |

**我们当前处于 L4 层级**，这对于"人工设计 + AI 辅助生成 + 人工微调"的工作流来说是合理的起点，但存在明确的提升空间。

## 二、当前体系的优势（做对了什么）

### 2.1 物理度量体系（PhysicsMetrics.cs）— 行业标准做法

项目已建立了**全局物理常量中心**，所有跳跃极限、间隙安全值、碰撞体尺寸均从运动学公式精确推导：

- `MAX_JUMP_HEIGHT = v²/(2g) = 20²/(2×80) = 2.5 格` — 公式正确
- `MAX_JUMP_DISTANCE = 4.5 格`，含 coyote time = 5.85 格 — 公式正确
- 安全值 vs 极限值的双档设计（safe=4/2, limit=5.85/2.5）— 符合行业惯例

> **行业对标**：error454 的 Platformer Physics 101 使用完全相同的运动学方程。Cloudberry Kingdom 也是从物理参数预计算跳跃极限。**此处完全符合行业标准。**

### 2.2 视碰分离（SpriteAutoFit + PhysicsMetrics）— 行业最佳实践

碰撞体尺寸在 PhysicsMetrics 中定义，视觉表现通过 SpriteAutoFit 自适应，换肤不影响物理。

> **行业对标**：Celeste 开发者 Maddy Thorson 明确分享过"角色碰撞体小于视觉 Sprite"的设计。**此处符合行业最佳实践。**

### 2.3 JumpArcVisualizer — 优秀的设计师辅助工具

在 Scene 视图中实时绘制跳跃抛物线（含半重力顶点效果），设计师可以直观判断间隙/高台是否可跨越。

> **行业对标**：Cloudberry Kingdom 的核心方法就是"预计算所有可能的跳跃目的地"。JumpArcVisualizer 是这一理念的可视化版本。**此处是行业优秀实践。**

### 2.4 ASCII 字符映射体系 — 与 VGLC 对齐

项目的字符映射覆盖了 18 种元素类型，与 VGLC（Video Game Level Corpus）的 SMB 标准格式在理念上一致：

| 对比项 | VGLC SMB | MarioTrickster | 评价 |
|--------|----------|----------------|------|
| 空气 | `-` | `.` | 等价，`.` 更直观 |
| 地面 | `X` | `#` | 等价 |
| 敌人 | `E` | `E`/`e` | 更细分（弹跳怪/简单敌人） |
| 金币 | `o` | `o` | 一致 |
| 管道 | `<>[]` 多字符 | 无 | 暂不需要 |
| 出生点/终点 | 无（元数据） | `M`/`G` | 更实用 |

### 2.5 防重登记簿体系 — 超越行业常见做法

TEMPLATE_REGISTRY.md 的"语义指纹 + 黑名单 + 灵感池"体系，在 AI 辅助关卡生成领域是非常前沿的实践。MarioGPT 论文中只用了 Novelty（编辑距离）来衡量新颖性，而本项目的语义级去重更加精准。

## 三、当前体系的差距（需要改进什么）

### 3.1 【严重】文档间物理约束数值不一致

这是**最严重的系统性问题**。同一个物理约束在不同文档中有不同的数值，直接导致 AI 和人类设计师生成不合规的模板：

| 文档 | 水平极限 | 垂直极限 | 状态 |
|------|---------|---------|------|
| **PhysicsMetrics.cs**（真相源） | safe=4, limit=5.85 | safe=2, limit=2.5 | ✅ 正确 |
| **PHYSICS_METRICS_GUIDE.md** | safe=4, limit=5.85 | safe=2, **极限=3** | ❌ 过时 |
| **AI_PROMPT_WORKFLOW.md** | safe=4, limit=5 | safe=2, **极限=2.5** | ✅ 已修复(S43) |
| **ASCII_Templates.md** | safe=4, limit=5 | safe=2, **极限=2.5** | ✅ 已修复(S43) |
| **LevelStudio_DesignGuide.md** | **极限=4** | **极限=3** | ❌ 严重过时 |
| **MarioTrickster_Testing_Guide.md** | "超过4格宽间隙" | **"3格高平台"** | ❌ 过时 |

> **根因**：PhysicsMetrics.cs 在 S32 建立时就是正确的（2.5格），但 LevelStudio_DesignGuide.md 和 Testing Guide 是更早期（S26b-S29）编写的，当时可能用的是估算值"3格"，后来从未同步更新。S43 已修复了 AI_PROMPT_WORKFLOW.md 和 ASCII_Templates.md，但另外两个文档仍然过时。

**影响**：
- 人类设计师看 LevelStudio_DesignGuide.md 会按"3格"设计 → 实际超出 2.5 格极限
- AI 如果被喂了 LevelStudio_DesignGuide.md 的内容也会按错误约束生成
- 测试指南中的预期结果与实际验证器行为不一致

### 3.2 【严重】LevelStudio_DesignGuide.md 声称"空格也视为空气"

第 46 行：`| ' ' | 空格也视为空气 | 不生成 |`

这与所有提示词文档中"绝对禁止空格，必须用 `.`"的要求矛盾。虽然 AsciiLevelGenerator 的代码确实将空格视为空气（不生成任何物体），但这会导致：
- 行宽不一致（空格在行尾会被编辑器自动裁剪）
- 验证器的间隙检测可能出错（空格 vs 点号的处理差异）
- 从外部 AI 获取的模板如果包含空格，粘贴后可能产生不可预测的结果

### 3.3 【中等】验证器仍为启发式检查（L4），缺乏路径连通性验证

当前验证器的核心逻辑是：
1. 逐行扫描间隙宽度
2. 逐列扫描高台高度差
3. 检查 M/G 存在性

**缺失的关键检查**：

| 缺失项 | 说明 | 行业做法 |
|--------|------|---------|
| **路径连通性** | 不验证从 M 到 G 是否存在可达路径 | Cloudberry Kingdom: Player AI 跑通；MarioGPT: A\* 搜索 |
| **动量/助跑因素** | 不区分"静止起跳"和"全速助跑起跳"的跳跃距离差异 | Cloudberry Kingdom: 考虑玩家状态 |
| **多路径分析** | 不识别上下分支路径（如 T07），可能误判某条路径为死路 | 图搜索天然支持 |
| **垂直下落关卡** | 不验证下落路径的安全性（如 T08 的竖井） | 需要方向感知的可达性分析 |
| **弹跳怪/弹跳平台辅助** | 不考虑踩弹跳怪/弹跳平台获得额外高度的情况 | 需要元素交互感知 |

> **评估**：对于当前"人工设计 + AI 辅助 + 人工微调"的工作流，L4 启发式检查配合 JumpArcVisualizer 的人工验证是**够用的**。但如果未来要实现"全自动关卡生成流水线"（路线 A 的终极目标），则需要升级到至少 L2（A\*/BFS 可达性验证）。

### 3.4 【中等】8 个 ASCII 模板中部分存在物理可疑设计

用 Python 脚本对 8 个模板进行静态分析，发现以下可疑点：

| 模板 | 问题 | 严重程度 |
|------|------|---------|
| T01 地刺深渊 | 第 3 行 `=` 到 `=` 间距约 17 格（远超极限），但中间有第 8/10 行的踏脚石 | 低（设计意图） |
| T04 火焰走廊 | 第 5 行 `M....P.....P.....P.....G.....` 宽 30 字符但只有 9 行高，走廊高度仅 4 格 | 低（设计意图） |
| T07 踩敌跳板 | 下路弹跳怪间距 4 格，需要踩踏获得额外高度才能跨越 | 中（依赖弹跳怪交互，验证器无法检测） |
| T08 垂直坠井 | 伪装墙 `F` 下方是地刺 `^`，玩家穿过后直接死亡 | 低（设计意图：视觉欺骗） |

> **评估**：这些大多是有意的设计，但验证器无法区分"有意的极限设计"和"无意的物理死路"。建议在模板中添加设计意图注释。

### 3.5 【低】提示词中缺少关键设计约束

对比行业最佳实践，当前提示词（AI_PROMPT_WORKFLOW.md）缺少以下约束：

| 缺失约束 | 说明 | 行业来源 |
|----------|------|---------|
| **最小平台宽度** | 玩家需要至少 2 格宽的平台才能安全落脚 | VGLC 数据分析：SMB 中最小平台宽度为 2 |
| **间隙后必须有落脚点** | 跨越间隙后必须有足够宽的安全平台 | Cloudberry Kingdom: 预计算跳跃目的地 |
| **敌人不应放在极限跳跃落点** | 极限跳跃的落点处不应有敌人，否则玩家无法同时处理跳跃精度和敌人 | GMTK 关卡设计理论 |
| **教学渐进原则** | 新机制应先在安全环境中出现一次，再在危险环境中使用 | Nintendo 内部设计原则（Miyamoto 四步法） |
| **视觉引导** | 金币/收集物应形成引导线，暗示正确路径 | 行业通用 |

## 五、总结评价

### 当前体系评分

| 维度 | 评分 | 说明 |
|------|------|------|
| 物理度量体系 | ⭐⭐⭐⭐⭐ | 从运动学公式精确推导，全局常量中心，行业标准做法 |
| 视碰分离 | ⭐⭐⭐⭐⭐ | 完全符合 Celeste 级别的行业最佳实践 |
| 设计师工具 | ⭐⭐⭐⭐ | JumpArcVisualizer 优秀，但需要人工使用 |
| 验证器 | ⭐⭐⭐ | 启发式检查够用，但缺乏路径连通性验证 |
| 提示词体系 | ⭐⭐⭐ | 覆盖了基本约束，但缺少部分关键设计原则 |
| 文档一致性 | ⭐⭐ | 多处文档与代码真相源不一致，是最大的系统性风险 |
| 防重体系 | ⭐⭐⭐⭐⭐ | 语义指纹 + 黑名单，超越行业常见做法 |

### 核心结论

> **验证器本身的逻辑没有根本性错误**（S43 修复后），问题的根源是**文档间物理约束数值不一致**导致 AI 和人类按错误的约束生成模板。这是一个**治理问题**而非**技术问题**。
>
> 当前的 L4 启发式验证 + JumpArcVisualizer 人工验证的组合，对于"人工设计 + AI 辅助 + 人工微调"的工作流是**足够的**。但如果要实现全自动关卡生成流水线，需要升级到 L2（BFS/A\* 可达性验证）。
>
> **最紧急的行动是统一所有文档的物理约束数值**（P0），这是投入产出比最高的改进。
