# S106 最小可行性升级包交付说明

**作者：Manus AI**

本次改造已经可以落地实现，且我已直接在 MarioTrickster 仓库内完成两份核心脚本的重构：`run_pipeline.py` 与 `blender_render_drive_video.py`。从实现结果看，你提出的 **S106 最小可行性升级包** 不只是“理论上可行”，而是已经被落成到主管线代码层。当前版本的核心目标，是把原先导致 **Semantic Silhouette Mismatch** 的几个底层放大器同时拆掉：极端留白、横版构图、缺少帽子/武器几何占位、整帧误抠、坏帧无熔断、颜色回正误伤，以及图集 pivot 不稳定。

这次重构后的管线，已经从“让模型凭空猜一个远景细狗”改为“先在几何与构图层强行给出稳定轮廓，再让生成模型沿着被约束的 silhouette 填充”。这正是你描述的修复方向，也是现阶段通用视频/动画模型在工业落地中更可靠的做法。

| 模块 | 已实现内容 | 目的 |
|---|---|---|
| Blender 代理体 | 增加 Head/RightHand 骨骼识别；自动生成高帽圆锥占位与长柄武器圆柱占位；代理体参与包围盒计算 | 把帽子与锤柄从“语义猜测”改成“物理轮廓先验” |
| Blender 构图 | `ortho_padding` 默认回调为 `1.6`，且 Proxy 接管时被限制在 `1.5 ~ 1.7` | 恢复主体有效像素占比，避免画面 95% 留白 |
| ComfyUI / 生成参数 | 默认切到 `512x768`、`steps=24`、`cfg=4.2`；S106 强制正反向提示词注入；知识护栏不再把这些默认值回退 | 解决长帽与长柄武器被横向裁切或缩成噪点的问题 |
| 参考图预处理 | 自动清理参考图左下角高风险签名/水印区，再上传参与生成 | 减少签名残留被模型二次学习 |
| 去背景 | 改成 Alpha/RGB 分离式掩码提取；当 Alpha 全不透明时自动退回 RGB 差分 | 避免整张帧被误判为前景 |
| 坏帧熔断 | 新增最小前景面积阈值；坏帧不参与全局 bbox 统计、颜色回正与关键计算 | 阻止蒸发帧把整套图集坐标系统拖崩 |
| Auto-Trim / 对齐 | 使用健康帧求全局 bbox；输出 Bottom-Center 对齐；统一底边加 5px 安全边 | 适配 Unity 角色落地感和脚底稳定性 |
| 元数据 | `pivot` 强制写为 `{x:0.5, y:0.0}`；逐帧写入 `is_bad`、`foreground_ratio`、`bbox` | 便于 Unity 与后续 QA 使用 |
| 颜色防御 | 逐帧安全均值平移；当主体面积不足时跳过颜色匹配；最终成图阶段不再全局误伤 | 避免把坏帧噪点拉亮成横向灰带 |

从工程角度看，本次实现已经覆盖了你要求的五个底层重构方向，而且做了一个额外的防弹修补：**当输入帧 Alpha 通道整张全不透明时，不再盲信 Alpha，而是自动切回 RGB 背景差分**。这是冒烟测试里实际暴露出来的问题；如果不修，很多普通 RGBA 帧会被误识别成“整张都是前景”，坏帧检测就会失效。因此我已经把这个坑一并补上了。

## 已落地的关键实现说明

`blender_render_drive_video.py` 现在不再只是“骨架圆柱代理体”。它会优先寻找 `Head` 或 `mixamorig:Head`，在头顶附加一个圆锥体 Dummy Hat；同时寻找 `RightHand` 或常见右手命名，在手部方向与重力方向混合后生成一个长圆柱 Dummy Hammer。这样做的意义在于，最终进入相机包围盒统计的不是人体裸骨，而是带帽子和长柄外轮廓的“可控 silhouette”。因此，相机自动构图得到的正交尺度会天然为帽子和锤柄留出生存空间，而不是让模型在大面积空背景里自行脑补。

`run_pipeline.py` 则被重构为 S106 风格的保护性主管线。它会先注入竖版分辨率、高步数与强制正反向提示词，再对参考图执行左下角污染清理。后处理阶段会给每帧计算前景占比与 bbox；若占比低于阈值，就把该帧标成坏帧。只有健康帧才参与全局裁切边界的求解，坏帧则被隔离，防止它把整套坐标轴和颜色基准拖偏。最后输出的 JSON 里，也会把坏帧信息和统一 pivot 一并写出，方便 Unity 或后续脚本继续接管。

## 已完成验证

我已经完成两层验证。第一层是 Python 语法检查，`run_pipeline.py` 与 `blender_render_drive_video.py` 均可正常编译。第二层是一个不依赖外部 ComfyUI 服务的本地冒烟测试：我构造了带左下角污染块的参考图，以及“一帧正常主体 + 一帧极小蒸发主体”的测试输入，随后跑通参考图清理、去背景、坏帧熔断、像素化、逐帧颜色回正、图集拼装与元数据写出。最终测试结果显示：坏帧被正确识别、健康帧参与 bbox 统计、`pivot` 被正确写为底部中心，且逐帧颜色防御会自动跳过坏帧。

## 对你附加建议的结论

你的附加建议不仅正确，而且应当被视为 **主生产线方向**。也就是说，**S106 可以作为当前 AI 视频链路的止血升级包与实验稳定器，但不应被误判为最终商业主线**。原因很简单：Dummy Props 能显著提升“画出来”的概率，却无法从根本上保证长杆武器在长时间轴上的刚体稳定。通用扩散/视频模型对细长刚体的高频轮廓一致性仍然很弱，这会自然表现为锤柄弯折、抖动、粗细漂移与空间相位错乱。

因此，真正适合商业闭环的方案，仍然是你提出的 **Weapon Decoupling**：让 AI 管线专注于空手角色本体的呼吸、布料、皮肤与光影细节；把巨锤作为独立 Sprite 挂到 Unity 手骨节点下，由引擎负责刚体位姿与层级控制。这不是对 S106 的否定，而是对它的正确定位：**S106 负责把现有全自动链路从“必炸”拉回“可测、可控、可迭代”，而主生产线应转向 2D 骨骼分件与引擎装配。**

## 建议你的下一步执行顺序

接下来最合理的动作，不是继续无上限堆模型，而是先在现仓库里跑一次真实素材验证，重点观察四个指标：第一，Proxy 渲染后的主体像素占比是否明显上升；第二，ComfyUI 输出里帽子和锤柄的缺失率是否下降；第三，坏帧标记数量是否集中在少数异常帧而不是大面积蔓延；第四，Unity 导入后底边 pivot 是否稳定。若这四项达标，S106 就算完成了“止血升级”的目标。随后就可以开始拆分主生产线：把武器剥离成独立 Sprite，把现有 AI 动画段改为纯角色空手动画资产生成管线。

## 结论

结论非常明确：**可以实现，而且已经实现到代码层。** 这版 S106 并不能把通用视频模型直接变成真正的游戏工业长兵器动画解算器，但它足以把你当前这条会因 silhouette 失真而连续级联崩溃的自动化链路，重构成一条更稳的、可监控的、可继续演进的实验生产线。同时，你给出的“AI 负责柔体，引擎负责刚体”的方向，应当作为后续主生产线的正式路线。
