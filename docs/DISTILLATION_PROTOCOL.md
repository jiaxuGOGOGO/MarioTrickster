# 蒸馏落库协议（DISTILLATION_PROTOCOL）

> 本协议用于规范 MarioTrickster 的教程蒸馏、规则落库、ComfyUI 出图与 Unity 资产落地闭环。它不是替换现有双轨架构，而是以 `prompts/PROMPT_RECIPES.md` 为主骨架，对**防遗漏、保留高价值重复、稳定传递到最终资产**三类问题进行增强。

## 1. 协议目标与适用范围

当前仓库已经拥有较强的双轨架构与 Unity 适配约束，但之前出现过“高价值规则已被读到、却未正式落入配方库”的情况。这说明风险不再主要来自大方向错误，而来自**最后一层蒸馏验收仍偏人工记忆**。本协议的目标，是把这层隐性风险改造成可核查、可提交、可回溯的显式流程。

| 目标 | 协议含义 | 与现有体系的关系 |
| :--- | :--- | :--- |
| **防遗漏** | 把“看过”与“入库”拆开管理，要求逐条回查 | 增强 `PROMPT_RECIPES.md`，不替代它 |
| **保留高价值重复** | 不把重复一概视为冗余，而把重复强度当成价值信号 | 保留现有有效规则，只补判定标准 |
| **稳定传递** | 要求规则从蒸馏一路带着证据走到 ComfyUI 与 Unity | 与 `ART_BIBLE.md`、风格保真文档协同 |

## 2. 外部实践中真正值得吸收的部分

GitHub 官方把 tasklist 描述为一种把工作拆成可追踪小项并逐条勾选完成的方式，这非常适合高遗漏成本的验收场景。[1] GitHub 同时支持仓库级 PR 模板，用来在提交入口强制填写结构化信息。[2] 这两点并不要求仓库照搬外部流程，但它们提供了一个非常有用的结论：**蒸馏工作不应只写“总结”，而应留下逐条可核查的入库清单与结构化提交说明。**

Reddit 关于“如何在长内容总结中不遗漏重点”的讨论，反复强调先分块、再汇总、再回查，而不是一次性压缩整份长文档。[3] GitHub 社区关于管理长讨论的经验也说明，真正值得保留的往往不是“唯一被提到的信息”，而是那些被反复验证、需要 recap、需要沉淀进主文档的高频约束。[4] 迁移到本仓库后，可以直接转译为两条原则：第一，教程蒸馏必须遵守“分块抽取 → 合并回查”；第二，**重复出现本身可以是价值信号，而不一定是冗余噪声。**

## 3. 与现有仓库主骨架的兼容方式

本协议明确接受并延续现有双轨架构：上半部技法库负责 How，下半部实体库负责 What，二者在 ComfyUI 中十字交叉，并通过节点物理隔离避免语义缠绕。它也继续接受仓库已有的最高优先级，即 `物理防滑步 > 空间透视正确 > 光影材质表现`。因此，任何新吸收的做法都只是一层**验收与落库强化壳**，而不是另起一套并行体系。

| 不变项 | 继续保留的原因 | 本协议只新增什么 |
| :--- | :--- | :--- |
| **双轨分治** | 已经能把教程语义与实体蓝图有效拆离 | 新增“每条规则必须有归属” |
| **节点物理隔离** | 已经能支撑 ComfyUI 稳定落点 | 新增“每条新增规则必须写明节点落点” |
| **Unity 适配优先** | 已经锁住横版、Pivot、PPU 与防滑步红线 | 新增“最终必须在 Unity 阶段留痕验收” |

## 4. 强制入库核查清单

> 只要这份清单还有未勾项目，就不能宣布蒸馏完成。[1]

- [ ] 已完成按章节、主题或技法簇的分块抽取。
- [ ] 已把每个分块转成可执行规则，而不是停留在解释性文字。
- [ ] 已为每条规则指定双轨归属与节点落点。
- [ ] 已对高风险规则进行第二遍回查，尤其是动作纠偏、透视边界、材质例外、风格漂移抑制项。
- [ ] 已识别并标注本次的**核心规则**与**高价值重复**。
- [ ] 已区分纯重复与真正冗余，只删除完全等价且无新增信息的内容。
- [ ] 已生成 recap，明确写出 `新增规则 / 高价值重复 / 已剔除冗余 / 待确认疑点`。
- [ ] 已完成仓库落库、记录更新、提交与远端推送确认。

## 5. 高价值重复与冗余的判定口径

过去最容易出错的地方，并不是看不见重复，而是把所有重复都当成应该压缩的噪声。实际上，重复可以携带“高频验证”“常见失败模式”“风险强调”这三类额外价值。[3] [4] 因此，本协议把重复拆成三类，而不再只分“保留 / 删除”两档。

| 类型 | 判定标准 | 处理方式 |
| :--- | :--- | :--- |
| **核心规则** | 同一规则在多个教程中出现 ≥2 次，且核心表述一致 | 升级进主规则或验收项，禁止删除 |
| **高价值重复** | 规则虽重复，但补充了失败模式、边界条件、适用例外或更清晰落点 | 合并信息，不删除约束 |
| **纯重复** | 语义、边界、节点落点、影响范围完全一致 | 保留最完整表述，删除其余重复文本 |

只有在**完全等价且无新增信息**时，合并才被视为合法去冗余。只要新增了任何约束、场景边界、失败表现或数值范围，就不再是纯重复，而应被视为高价值重复。

## 6. 四段验收节点

为了保证蒸馏效果真的能传到最终资产，本协议把链路拆成四段，每一段都必须留下可检验输出物，而不能只说“应该没问题”。

| 阶段 | 必须有的输出物 | 通过标准 |
| :--- | :--- | :--- |
| **蒸馏** | 分块抽取记录、遗漏风险清单、recap | 能证明教程已被逐块检查，而不是整体压缩 |
| **落库** | `PROMPT_RECIPES.md` / 协议文档 diff、规则归类、节点落点 | 能证明规则已进入双轨主库，而不是停留在临时笔记 |
| **出图** | 可复制的 ComfyUI 节点配置图纸、关键参数、锚点对照或抽样验证 | 能证明规则可被节点执行，而不是只停留在文案层 |
| **Unity** | 导入设置确认、Pivot/PPU/FitMode 对齐结果、资产路径或验收记录 | 能证明最终资产仍服从横版与防滑步红线 |

## 7. 提交与回溯要求

GitHub 的模板化入口之所以有价值，不在于“样式整齐”，而在于它把提交说明从随手备注提升为结构化证据。[2] 因此，本仓库后续凡是涉及蒸馏落库的 commit 或 PR，都必须附带“本次蒸馏入库清单”，至少写清四类字段：`新增规则`、`高价值重复`、`已剔除冗余`、`待确认疑点`。如果没有这段结构化清单，就无法从 Git 历史中高效判断某次提交到底改变了什么，也无法复盘是否存在“读到但没入库”的断层。

## 8. 执行结论

本协议的核心不是“让文档变多”，而是让**每一条高风险规则都拥有从教程到配方、从配方到节点、从节点到 Unity 的可追踪路径**。在 MarioTrickster 现有骨架已经基本正确的前提下，这种增强方式比“另起新体系”更稳妥，因为它只补足闭环最薄弱的一层，同时保留仓库已经验证有效的双轨结构与 Unity 适配原则。
