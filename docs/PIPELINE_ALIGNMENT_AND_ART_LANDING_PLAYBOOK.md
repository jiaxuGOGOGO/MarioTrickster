# MarioTrickster 防跑偏运营模型与美术落地回接路线图

> **目标**：在不牺牲关卡主线推进速度的前提下，把当前美术探索收束为可落地、可验证、可回接的资产生产系统，并确保未来美术跑通后能够无缝接回现有白盒关卡与运行时代码。

---

## 1. 当前项目真正的问题是什么

MarioTrickster 当前并不是“从关卡设计项目变成了美术项目”，而是**主线与支线的边界暂时失焦**。项目原始核心始终是 **Trickster/Mario 双角色平台解谜体验 + 可持续扩展的白盒关卡生产**，而最近连续推进的 LoRA、提示词、教程蒸馏、ComfyUI 动画与伪 3D 研究，本质上都属于 **美术供给能力建设**，不是新的产品主线。

问题真正出在三个层面。第一，**任务源混线**：原本应由“关卡需要什么”倒推“美术做什么”，后来阶段性变成了“美术还能继续探索什么”。第二，**完成定义混线**：教程蒸馏、配方补全、LoRA 登记、本地验证模板，这些都属于“能力准备完成”，但它们还不等于“游戏资产已经可用”。第三，**接回点没有明确冻结**：如果不提前定义“未来一张图、一套动画、一组场景素材到底要接到 Unity 的哪个槽位、以什么命名、遵守什么碰撞与 PPU 约束”，那么美术越做越多，反而越难回接。

| 现象 | 实际含义 | 风险 |
|---|---|---|
| 一直在蒸馏教程、研究 ComfyUI、登记 LoRA | 在补“生产方法”而不是直接做关卡 | 如果没有时间盒和出口条件，会误以为项目已经在前进 |
| 迟迟没有正式量产美术 | 说明风格锚点和验证闭环还没完成 | 继续做大规模出图只会增加返工 |
| 感觉关卡主线不见了 | 不是不存在，而是没有被重新设为唯一上游任务源 | 关卡、机制、美术会相互等待 |
| 担心以后接不回游戏 | 说明缺的是“资产接入规范”而不是更多创作灵感 | 越晚补，整合成本越高 |

因此，当前最优策略不是“继续同时把所有事都推进”，而是要把项目改成一个**双轨但单主线**的运行模型：**关卡白盒是唯一主线，美术是并行使能轨，美术是否继续推进必须由明确的接回目标驱动。**

## 2. 建议采用的项目运行模型：双轨并行，单主线统领

建议从现在开始，把项目固定为三条轨道，但只有一条是主线。**A 轨是关卡白盒主线，B 轨是美术使能支线，C 轨是资产回接桥接层。** 这三条轨道都重要，但优先级和完成标准必须完全不同。

| 轨道 | 定位 | 负责人视角 | 是否允许阻塞其他轨道 | 核心产物 |
|---|---|---|---|---|
| **A 轨：关卡白盒主线** | 唯一产品主线 | 先把玩法、节奏、机关关系、路线可读性做对 | **不允许被 B 轨阻塞** | ASCII 模板、片段库、白盒关卡段、可玩验证结果 |
| **B 轨：美术使能支线** | 为未来量产与换肤做准备 | 解决风格统一、角色一致性、批量生产、动画/伪3D方法 | **只能服务 A 轨，不得反向劫持节奏** | 概念锚点、LoRA 验证结论、工作流模板、参数甜区、资产样张 |
| **C 轨：回接桥接层** | 保证未来能装进游戏 | 规定命名、尺寸、PPU、Pivot、Sprite 槽位、分层与导入规则 | 既不能缺席，也不能过度设计 | 资产规范、目录结构、接入清单、替换规则 |

这意味着你以后每做一件事，都先问一句：**它属于 A、B、还是 C？** 如果一件事不属于任何一轨，或者不能证明它会在两周内产出该轨道的明确成果，那它就应该暂停。

## 3. 不跑偏的核心原则

### 3.1 关卡优先原则

以后所有新任务，首先必须回答一个问题：**它是否直接帮助你更快做出更好的关卡段？** 如果答案是否定的，那么它只能进入 B 轨的“限时探索箱”，不能占用主工作时间。

具体执行上，建议把每周有效项目时间按 **70/20/10** 切分。约 **70%** 时间固定投入 A 轨，也就是继续用 `Level Studio / Custom Template Editor / Build From Text` 做白盒拼装、试玩、修节奏、补片段；约 **20%** 时间投入 B 轨，专门推进美术生产中最短板的一步；剩余 **10%** 用于 C 轨，做真正必要的接回规范和目录清理。这个比例的核心不是精确计时，而是强制提醒自己：**美术探索不能重新变成默认主线。**

### 3.2 美术只做“已命名需求”

未来任何美术探索任务，都必须先绑定一个**已命名的资产需求**，不能再做“泛泛试试看”。例如，不做“研究一下角色动画”，而是做“为 Trickster 站立、向右移动、起跳、落地四个状态建立首套可接入白盒的参考资产规范”；不做“研究伪 3D”，而是做“为横版场景背景建立一套可用于 Parallax 的三层切分模板”。

| 错误任务写法 | 正确任务写法 |
|---|---|
| 继续蒸馏更多动画教程 | 把现有蒸馏内容重写成 `Trickster 头部轻摆 / 短循环披风摆动 / 场景镜头平移` 三张工作流任务卡 |
| 研究如何批量出图 | 为 `Ground / Platform / Spike / HiddenPassage / Checkpoint` 五类环境件建立首套量产模板 |
| 看看伪 3D 能做到什么程度 | 为 `前景遮挡 / 主层可玩区 / 背景远景` 建立可导入 Unity 的三层场景包规范 |

### 3.3 任何美术成果都必须有“接回定义”

从现在开始，美术成果只有满足“可回接定义”才算真正完成。这个定义至少应包含五项：**目标槽位、目录位置、命名规则、导入参数、废弃条件**。只要少了一项，这个成果就还只是实验样张。

| 字段 | 必填说明 |
|---|---|
| **目标槽位** | 这张图最终替换 Unity 中哪个位置，例如 `LevelThemeProfile.groundSprite`、`platformSprite` 或角色某动画状态 |
| **目录位置** | 进入仓库或本地生产目录的固定路径，例如 `Assets/Art/Styles/Style_Pixel_16Bit/Environment/Ground/` |
| **命名规则** | 例如 `ground_grass_a_idle_v01.png`、`trickster_run_sheet_v02.png` |
| **导入参数** | PPU、Pivot、是否切片、是否需要 `AI_SpriteSlicer`、是否要求透明边裁切 |
| **废弃条件** | 出现风格漂移、比例不匹配、轮廓污染、动画闪烁时直接判废，不进入量产库 |

## 4. 你现在最该做的结构性调整

### 4.1 立即恢复 A 轨：白盒关卡继续向前，不等待最终美术

你现在最应该做的第一件事，不是继续扩写美术文档，而是**马上恢复 1~2 个完整关卡段的白盒迭代**。原因很简单：关卡主线是上游需求源，只有白盒继续生长，后面的美术才知道哪些资产真的需要，哪些只是脑补需求。

建议直接以现有的 `Level Studio / Custom Template Editor / Build From Text` 为入口，给自己一个非常窄的短冲刺目标，例如“做一个以伪装切换 + 单向平台 + 检查点为核心的完整教学段”和“做一个以 Trickster 操控道具 + 隐藏通道为核心的完整挑战段”。这两段不需要美术先到位，依然可以用现有白盒验证节奏、信息量与失败反馈。

### 4.2 把 B 轨从“大研究”收束成“单一瓶颈突破”

当前 B 轨最大的现实瓶颈并不是“ComfyUI 还不够强”，而是 **`trickster_style` 本地验证结果还没出来**。所以接下来所有美术动作都应该围绕这一点收束，而不是继续发散。

只要 `MarioTrickster_Style_epoch_10.safetensors` 的本地验证没有完成，就先不要进入正式量产，也不要轻易扩展到更多 LoRA、更多教程蒸馏或更复杂的动画流。因为一旦基础风格锚点都没锁住，后续所有角色、道具、场景和动画都会产生叠加返工。

### 4.3 让 C 轨提前变成“薄桥”，不要最后一次性补课

很多项目会在美术跑通之后才想“怎么导回游戏”，这会非常痛苦。你的项目其实已经有很好的基础，因为现有文档已经明确了 **视碰分离、PPU/Pivot 统一、`LevelThemeProfile` 槽位替换、多风格目录结构**。现在需要做的不是新造一套系统，而是把桥做薄、做清楚。

建议只补三类最小桥接件：第一，**资产槽位清单**；第二，**目录和命名规则**；第三，**首批可替换对象名单**。不要现在就大谈全局运行时换肤，不要重构资源系统；只要先确保未来第一批美术能替换进来不炸就够了。

## 5. 推荐的阶段门（Stage Gates）

为了避免“感觉做了很多，但没有一个东西真的完成”，建议把当前项目接下来固定成五个阶段门。每个阶段门都必须满足进入条件和完成条件，不能跳步。

| 阶段门 | 目标 | 进入条件 | 完成条件 |
|---|---|---|---|
| **Gate 0：白盒续航恢复** | 重新让关卡主线持续产出 | 已确认关卡主线不等待美术 | 至少新增或迭代 1~2 个可玩的白盒关卡段 |
| **Gate 1：风格锚点锁定** | 锁定项目美术方向 | 已有概念锚点蓝图与 LoRA 候选 | `trickster_style` 本地验证完成，并得出推荐权重/触发词/负面词 |
| **Gate 2：最小量产包** | 做出第一批真正可接入资产 | Gate 1 完成 | 角色/环境/道具至少各有一组通过接回定义的正式资产 |
| **Gate 3：动画与伪3D模板化** | 把研究变成可重复工作流 | Gate 2 完成，至少一套静态资产已稳定 | 四条窄工作流中至少两条形成模板并复现成功 |
| **Gate 4：游戏内回接** | 资产真正进入 Unity 运行 | Gate 2 或 Gate 3 有可用结果 | 通过 `LevelThemeProfile` 或现有导入链路替换进测试场景，并完成最小视觉验收 |

这五个门里，**Gate 0 和 Gate 1 必须并行，但 Gate 2 以后必须建立在前两者都过关的基础上。** 这样你就不会再在“白盒还没稳定”时急着追动画，也不会在“LoRA 还没验证”时急着大批量出图。

## 6. 未来四周的最优行动顺序

下面这套顺序，目标就是让项目既不偏航，也不空转。

| 周次 | A 轨：关卡白盒 | B 轨：美术使能 | C 轨：桥接层 |
|---|---|---|---|
| **第 1 周** | 恢复 1 个教学段白盒迭代 | 完成 `trickster_style` 本地验证首轮扫描 | 列出首批 10 个最需要未来替换的资产槽位 |
| **第 2 周** | 再完成 1 个挑战段白盒迭代 | 根据验证结果修正 LoRA 登记与提示词结构 | 建立首批目录/命名规范并绑定到槽位 |
| **第 3 周** | 用白盒验证新增机关是否真的需要新美术 | 做第一批角色/环境/道具正式样张 | 用测试场景尝试替换 3~5 个对象 |
| **第 4 周** | 继续打磨关卡节奏和信息量 | 从四条窄工作流里挑两条做第一次模板化复现 | 把成功接入流程写成标准作业文件 |

这里的关键是：**每周都有 A 轨成果，每周 B 轨只打一处瓶颈，每周 C 轨只补最必要的一层桥。** 这会让项目同时保持前进感和可控性。

## 7. 什么事情现在应该停止或降级

为了防止重新失焦，下面这些事情建议马上降级，不要再默认持续推进。

| 事项 | 建议处理方式 | 原因 |
|---|---|---|
| 无限继续蒸馏新教程/新书 | **暂停为默认动作**，仅在现有工作流确实缺关键知识时再补 | 你现在缺的不是更多“知道”，而是把已知内容转成模板 |
| 在 LoRA 未验证完成前就批量出正式资产 | **禁止** | 会把未稳定风格扩散到整个素材库 |
| 追求一个万能动画总工作流 | **暂停** | 现阶段应坚持四条窄工作流分治，而不是做大而全神流 |
| 为未来多风格切换做重型系统设计 | **降级为文档约束** | 当前已有 `LevelThemeProfile` 和视碰分离基础，先满足首批接入即可 |
| 因美术未完而暂停关卡设计 | **禁止** | 关卡才是需求源，不继续做关卡就无法判断真正需要哪些资产 |

## 8. 资产真正落地时的接回标准

为了保证“以后真的能接上”，建议把正式资产的验收标准固定为三层：**风格层、工程层、玩法层**。任何一层不过，都不算真正完成。

| 验收层 | 必须回答的问题 | 不通过时的处理 |
|---|---|---|
| **风格层** | 它是否命中项目既定画风？是否存在轮廓污染、题材污染、动画闪烁？ | 回到 LoRA / Prompt / Control 条件重新生成 |
| **工程层** | 它是否符合 PPU、Pivot、尺寸、切片、命名、目录、透明边规范？ | 禁止直接导入关卡，先做工程修整 |
| **玩法层** | 它进入场景后是否仍然保持碰撞、可读性、前后景关系和交互辨识度？ | 视为“美术正确但游戏不适配”，必须重新调视觉层 |

这套三层验收尤其重要，因为很多美术图在图像层面“很好看”，但一进游戏就会出现三类问题：看不清可交互物、视觉体积与碰撞不匹配、运动中信息噪声过多。你的项目已有视碰分离架构，因此完全可以把这些问题控制在视觉层，而不必回头改物理逻辑。

## 9. 一句话决策规则

为了让你以后在具体工作中少纠结，可以把下面几条一句话规则当作日常判断器。

| 情况 | 直接决策 |
|---|---|
| 美术任务不能说明最终替换哪个对象 | 先不做 |
| 美术探索会阻塞本周白盒关卡推进 | 暂停该探索 |
| LoRA 还没验证完却想批量出图 | 不允许 |
| 研究内容无法写成参数表/模板/故障树 | 说明它还只是笔记，不算生产能力 |
| 资产进游戏后影响碰撞或玩法判读 | 只改视觉接入，不改物理真理源 |
| 不知道该优先做什么 | 永远先回到 A 轨，继续关卡白盒 |

---

## 10. 结论

MarioTrickster 现在最需要的不是再扩一条新管线，而是**把已有能力重新排队**。项目的正确结构应当是：**关卡白盒持续产出需求，美术使能围绕单一瓶颈逐个突破，桥接层提前把未来接入规则做薄做清楚。**

只要你坚持三个动作，项目就不会再乱走偏。第一，**关卡白盒每周必须继续增长**；第二，**美术只围绕已命名资产需求推进，并优先打通 `trickster_style` 验证闭环**；第三，**所有正式资产都必须先通过接回定义，再允许进入游戏**。这样做的结果是：即使美术系统还在迭代，项目也不会停摆；而一旦美术跑通，它又能非常自然地接回当前的关卡、系统和运行时代码。
