# MarioTrickster 策划完整使用指南

> **适用对象**：游戏策划、关卡设计师、非程序背景的协作者
>
> **核心原则**：你只需要用大白话描述目标，系统和 AI 会在后台处理所有技术细节。本指南帮你搞清楚"哪些能随便动、哪些要小心、哪些绝对不能碰"。

---

## 一、项目当前状态一句话总结

MarioTrickster 是一款非对称对抗平台跳跃游戏。Mario（闯关者）要跑到终点，Trickster（捣蛋者）要伪装成关卡道具暗中阻碍 Mario。项目的**玩法骨架和关卡生产工具已经完备**，当前主要缺的是正式美术资产和音效。你现在就可以开始做关卡、验证节奏、打磨博弈体验。

## 二、操作边界警示系统

### 🔴 红线（绝对禁止 — 碰了就炸）

红线操作会直接导致**全部关卡崩溃、自动化测试失效或核心玩法逻辑断裂**。项目内置了 `RedLineGuard` 自动巡检系统，会在保存场景和进入 Play 模式时自动检测违规并尝试修复，但最好的做法是从一开始就不要碰。

| 编号 | 红线操作 | 为什么不能碰 | 如果你想达到类似目的，正确做法是 |
|------|----------|-------------|-------------------------------|
| R1 | 修改 `CELL_SIZE`（在 PhysicsMetrics.cs 中） | 这个值决定了"1个 ASCII 字符 = 多大的游戏世界"。改了它，所有关卡的平台间距、跳跃距离、碰撞体全部错位 | 调相机 Orthographic Size 或 Sprite PPU 来改变"看起来"的大小 |
| R2 | 缩放角色 Root 节点的 `localScale` | Root 缩放会连碰撞体一起缩，导致角色卡墙、穿地板、测试全部失败 | 只改 `Visual` 子节点的缩放，或改 Sprite 的 PPU |
| R3 | 手动修改 Mario/Trickster 的碰撞体 `size` 和 `offset` | 碰撞体故意比视觉小（宽容手感设计），改了会破坏所有平台间隙的可达性计算 | 如果确实需要改手感，修改 `PhysicsConfig` SO 并跑全套回归测试 |
| R4 | 在 Update/FixedUpdate 中写 `Instantiate()` | 每帧创建对象会导致严重卡顿甚至 GPU Timeout 崩溃 | 使用对象池（项目已有 LineRenderer 对象池模式可参考） |
| R5 | 绕过 `TricksterPossessionGate` 做第二套揭穿逻辑 | 五态门禁是 Trickster 的唯一真实状态链，绕过它会导致"看似揭穿但 Trickster 仍能操控"的体验 Bug | 所有揭穿都走 `ForceReveal()` 接口 |
| R6 | 删除或重命名 `ControllablePropBase` 的核心生命周期方法 | 所有可操控道具（地刺、封路机关、队列机关等）都继承它的 Telegraph→Active→Cooldown 状态机 | 如需扩展，用 `override` 覆写，不要删除基类方法 |
| R7 | 修改 ASCII 字符映射表中已有字符的含义 | 已有关卡模板和片段库都依赖这些映射，改了会导致旧关卡生成出错误元素 | 新增字符用未占用的符号，不要复用旧字符 |

**如果你不确定某个操作是否触碰红线**：直接问 AI，"我想做 XXX，这样做安全吗？"AI 会告诉你答案。

### 🟡 黄线（需要谨慎 — 能做但容易出错）

黄线操作本身是合法的，但如果操作不当容易引发连锁问题。**建议通过大白话告诉 AI，让 AI 代为执行**。

| 编号 | 黄线操作 | 风险点 | 正确流程 |
|------|----------|--------|----------|
| Y1 | 调整全局物理手感（跳跃高度、重力、移动速度） | 改了之后所有关卡的"能不能跳过去"标准都会变，旧关卡可能变成死局或太简单 | 修改 `PhysicsConfig` SO → 跑一遍 9-Stage 测试关卡 → 检查 JumpArcVisualizer 辅助线 → 确认无死局 |
| Y2 | 添加全新的游戏机制或机关类型 | 需要同时更新代码、ASCII 字典、调色板、测试和文档，漏一个就不完整 | 用大白话告诉 AI 你想要什么效果，AI 会一次性补齐所有配套 |
| Y3 | 大批量重命名美术素材 | 手动改名容易打错、漏改引用、破坏已有 Theme 绑定 | 使用 `策划生产助手`（Ctrl+T → Art & Effects Hub）的语义报告功能，确认后一键批量改名 |
| Y4 | 修改 `LevelThemeProfile` 的元素键名 | 键名和代码中的类型映射绑定，改错会导致换肤失败 | 只修改 Profile 中的 Sprite 引用（拖入新图片），不要改键名本身 |
| Y5 | 修改 `GameManager` 的回合逻辑或胜负判定 | 这是游戏流程的核心骨架，改错会导致无法正常结束回合 | 描述你想要的规则变化，让 AI 评估影响范围后再改 |
| Y6 | 在子类中重新声明基类已有的 `protected` 字段 | 会产生 CS0108 编译警告，且子类和基类各自维护一份数据导致逻辑混乱 | 直接使用基类的 `protected` 字段，不要用 `private` 重新声明 |
| Y7 | 修改 `SpriteStateAnimator` 的状态标签系统 | 当前使用字符串标签（"idle"/"run"/"jump"/"fall"），改为枚举会破坏已有测试和素材导入管线 | 如需新增状态，追加新字符串标签即可，不要改已有的 |

---

### 🟢 绿线（随意操作 — 策划的主战场）

以下操作完全安全，请尽情发挥创意：

| 编号 | 操作 | 说明 |
|------|------|------|
| G1 | 用 ASCII 文本拼关卡 | 在 Test Console 中写字符矩阵，点 Build 生成。改了不满意就清空重来 |
| G2 | 给白盒换美术素材 | 选中物体 → `Ctrl+Shift+A` → 拖入图片 → 应用。不会破坏任何逻辑 |
| G3 | 在 Scene 视图中移动/旋转已生成的关卡元素 | 微调位置完全安全，建议保持对齐整数网格 |
| G4 | 调整单个陷阱/平台的 Inspector 参数 | 比如某个地刺的伤害值、某个移动平台的速度、某个摆锤的摆动角度 |
| G5 | 使用 Test Console 的所有测试外挂 | 无敌、无冷却、无限能量、时间缩放等，不会影响正式版本 |
| G6 | 传送到关卡任意位置测试 | Teleport 功能随便用，不会破坏关卡状态 |
| G7 | 从片段库追加经典片段 | Tutorial Start、Trap Corridor、Final Sprint 等随意组合 |
| G8 | 创建新的 `LevelThemeProfile` | 右键 Create → MarioTrickster → Level Theme Profile，拖入 Sprite 即可 |
| G9 | 使用 SEF 视觉效果 | `Ctrl+Shift+Q` 给物体加描边、闪白、溶解等效果，可随时撤销 |
| G10 | 修改关卡的时间限制、生命数等数值 | 在 GameManager 或 PlayerHealth 的 Inspector 中调整 |
| G11 | 导入新的商业素材包 | 通过 Asset Import Pipeline (`Ctrl+Shift+I`) 导入，工具会自动分类 |
| G12 | 运行自动化测试 | `MarioTrickster → Run Tests → Export Full Report` 随时跑，看看有没有破坏什么 |

## 三、日常工作流操作手册

### 3.1 做关卡的完整流程

```
第一步：构思 → 第二步：ASCII 白盒 → 第三步：试跑调整 → 第四步：换美术 → 第五步：精调体验
```

**第一步：构思**

想清楚这段关卡要验证什么体验。比如："我想做一段让 Trickster 有两个伏击点可选的区域，Mario 需要观察哪个道具在动来判断 Trickster 藏在哪。"

**第二步：ASCII 白盒**

打开 Test Console (`Ctrl+T`)，在 Custom Template Editor 中写出关卡文本：

```
..............
..............
.M...^..^...G.
##############
```

上面这段表示：一条平地，起点 M 到终点 G 之间有两个地刺 `^`。

**第三步：试跑调整**

点击 Build From Text → 按 Play → 跑一遍。如果太简单就加障碍，如果跳不过去就缩短间距。**不要追求一次完美，快速迭代才是王道。**

**第四步：换美术**

白盒好玩了再换皮。选中白盒 → `Ctrl+Shift+A` → 拖入你的美术素材 → 应用。

**第五步：精调体验**

用 Teleport 反复测试关键节点，调整陷阱节奏、平台间距、敌人巡逻路线。

---

### 3.2 ASCII 字符速查表

| 字符 | 生成什么 | 颜色提示 | 常用场景 |
|------|----------|----------|----------|
| `.` | 空气（不生成任何东西） | — | 必须用点号填充空白，不能用空格 |
| `#` | 实心地面方块 | 棕色 | 地面、墙壁、天花板 |
| `M` | Mario 出生点 | 蓝色 | 每关必须有且只有一个 |
| `T` | Trickster 出生点 | 紫色 | 每关必须有且只有一个 |
| `G` | 终点 | 亮绿 | 每关必须有且只有一个 |
| `^` | 地刺陷阱 | 红色 | 碰到就扣血 |
| `~` | 火焰陷阱 | 橙色 | 周期性喷火 |
| `P` | 摆锤 | 棕色 | 来回摆动的大锤 |
| `=` | 移动平台 | 灰色 | 会沿路径移动 |
| `B` | 弹跳平台 | 黄色 | 踩上去弹飞 |
| `-` | 单向平台 | 浅灰 | 只能从下往上穿过 |
| `C` | 崩塌平台 | 橙色 | 踩上去会碎 |
| `>` | 传送带 | 蓝色 | 站上去会被推 |
| `E` | 弹跳怪 | 粉红 | 会跳跃的敌人 |
| `e` | 简单敌人 | 粉红 | 左右巡逻的敌人 |
| `F` | 伪装墙 | 蓝灰 | 看起来是墙但能穿过 |
| `H` | 隐藏通道 | 青绿 | 地下捷径入口 |
| `o` | 金币收集物 | 金色 | 可选收集品 |
| `[` | 封路机关 | 半透明蓝 | Trickster 可操控的临时墙 |
| `]` | 公开队列机关 | 橙红 | 有公开状态队列的机关 |

> **排版铁律**：
> - 空白区域**必须用 `.`（点号）填充**，绝对不能用空格
> - 保持完美矩形网格（每行字符数相同）
> - 第一行 = 最高层，最后一行 = 最低层

### 3.4 美术素材接入指南

**场景一：我有一张新的地刺图片，想替换白盒地刺**

1. 在 Scene 中选中那个白盒地刺（点击它）。
2. 按 `Ctrl+Shift+A`。
3. 把你的地刺图片拖到弹出窗口中。
4. 点击"应用"。
5. 完成。碰撞体、伤害逻辑、动画触发全部不变，只是外观换了。

**场景二：我买了一个商业素材包，里面有一整套角色动画**

1. 把素材文件命名好：`hero_idle_00`, `hero_idle_01`, `hero_run_00`, `hero_run_01`, `hero_jump_00`, `hero_fall_00`。
2. 选中角色（Mario 或 Trickster）。
3. 按 `Ctrl+Shift+A`，把整个文件夹拖进去。
4. 系统会自动识别 idle/run/jump/fall 状态并挂载动画。
5. 如果只有 `run` 帧也能用，其他状态会用静态图兜底。

**场景三：我有一张大合集图（很多小图拼在一起）**

1. 先用 AI Smart Slicer 裁切：`MarioTrickster → AI Smart Slicer`。
2. 或者用手动切片：在 Apply Art 中选择 Manual Grid 模式，指定行列数。
3. 裁切完成后再逐个应用到对应的白盒物体上。

**命名建议**：

| 素材类型 | 推荐命名格式 | 示例 |
|----------|-------------|------|
| 角色站立帧 | `角色名_idle_两位编号` | `mario_idle_00`, `mario_idle_01` |
| 角色奔跑帧 | `角色名_run_两位编号` | `mario_run_00`, `mario_run_01` |
| 角色跳跃帧 | `角色名_jump_两位编号` | `mario_jump_00` |
| 角色下落帧 | `角色名_fall_两位编号` | `mario_fall_00` |
| 环境循环动画 | `物体名_帧编号` | `torch_00`, `torch_01`, `water_00` |
| 静态道具 | `描述性名称` | `spike_red`, `platform_stone` |

## 四、关卡设计核心原则

### 4.1 体验护栏（设计时必须遵守）

| 原则 | 错误示范 | 正确示范 |
|------|----------|----------|
| Trickster 不能硬锁 Mario | 两条路都能被长时间封死 | 至少一条路保持可通过，或封锁只持续 2-3 秒 |
| 阻碍必须有预告 | 地刺突然弹出，完全没有反应时间 | 地刺弹出前闪红光 0.8 秒（Telegraph 阶段） |
| 被阻碍要给补偿 | Mario 被卡了但什么信息都没得到 | 被卡后 Mario 获得线索（残影、热度痕迹） |
| 重复恶心要递减 | Trickster 反复在同一个点卡人，零风险 | 同点重复操控会快速叠加暴露风险 |
| 失败不能螺旋 | 被卡一次后越来越难翻盘 | 每次被阻碍都推进 Mario 的反制进度 |

### 4.2 双路线设计建议

好的关卡应该给 Mario 至少两条路径选择，同时给 Trickster 有意义的伏击决策：

```
路线 A（快但危险）：
..........G.
....=...=...
..^...^.....
############

路线 B（慢但安全）：
............
............
.M..........
####..####..
....##....##
```

**设计检查清单**：
- Mario 有没有至少两条路可选？
- Trickster 能不能同时封死所有路？（不应该能）
- 每条路的风险/收益是否不同？
- Trickster 选择伏击点时有没有有意义的取舍？

## 五、测试与验证

### 5.1 快速自测流程

1. 做完关卡后，先自己跑一遍，确认 Mario 能到终点。
2. 再用 Trickster 视角想想："如果我藏在这里操控那个机关，Mario 还能过吗？"
3. 开启 God Mode 测试极端情况（比如所有陷阱同时触发）。
4. 用 Time Scale 0.5x 慢放观察陷阱节奏是否合理。

### 5.2 自动化测试

项目有 135+ 个自动化测试覆盖核心逻辑。你做完关卡后可以跑一遍确认没有破坏什么：

菜单：`MarioTrickster → Run Tests → Export Full Report (All)`

如果有测试失败，不要慌。看看失败的测试名称：
- 如果是物理相关的（跳跃高度、间隙宽度）：你可能不小心改了物理参数。
- 如果是状态机相关的：你可能不小心改了某个脚本。
- 如果是你完全没碰过的东西：可能是历史遗留问题，告诉 AI 处理。

## 六、常见问题 FAQ

**Q：我在 Scene 里拖动了一个平台，会不会出问题？**
A：不会。在 Scene 里移动已生成的物体完全安全。建议保持对齐整数坐标（方便后续维护）。

**Q：我不小心改了什么东西，游戏跑不起来了怎么办？**
A：按 `Ctrl+Z` 撤销。如果撤销不了，告诉 AI 你做了什么，AI 会帮你修复。

**Q：我想做一个目前游戏里没有的机关，怎么办？**
A：不要自己写代码。直接告诉 AI："我想要一个 XXX 效果的机关"，AI 会全套搞定。

**Q：美术素材导入后角色不动了/穿墙了怎么办？**
A：这通常是因为素材被错误地应用到了 Root 而不是 Visual。使用 `Ctrl+Shift+A` 重新应用，工具会自动处理。如果还是有问题，跑一次 `MarioTrickster → 红线巡检` 让系统自动修复。

**Q：我能不能直接在 Hierarchy 里复制粘贴关卡元素？**
A：可以，但建议用 ASCII 文本方式管理关卡布局（更容易修改和版本控制）。复制粘贴适合微调已有布局。

**Q：Test Console 里的作弊开关会影响正式版本吗？**
A：不会。所有作弊开关被 `#if UNITY_EDITOR || DEVELOPMENT_BUILD` 宏隔离，发布版本中完全不存在。

## 七、项目架构速览（了解即可，不需要记住）

| 模块 | 职责 | 策划需要关心吗？ |
|------|------|-----------------|
| Scripts/Ability | Trickster 的伪装、操控、能量系统 | 不需要，除非要调操控参数 |
| Scripts/Core | 基础框架（输入、碰撞、动画、关卡管理） | 不需要 |
| Scripts/Editor | 所有编辑器工具（Test Console、导入管线等） | 你每天都在用它们，但不需要看代码 |
| Scripts/Enemy | Trickster 和敌人的控制器 | 不需要 |
| Scripts/Gameplay | 博弈系统（怀疑度、热度、路线预算等） | 了解设计意图即可 |
| Scripts/LevelDesign | ASCII 生成器、物理度量、主题系统 | 不需要直接修改 |
| Scripts/LevelElements | 所有关卡元素（陷阱、平台、通道等） | 可以在 Inspector 中调参数 |
| Scripts/Player | Mario 的控制器和血量 | 不需要 |
| Scripts/UI | 游戏 UI | 不需要 |

## 八、与 AI 协作的最佳实践

### 说大白话，不说技术术语

| 不要这样说 | 要这样说 |
|-----------|----------|
| "帮我写一个继承 ControllablePropBase 的 WindTrap 类" | "我想要一个会吹风的机关，站上去会被推着走" |
| "把 jumpPower 从 20 改成 25" | "我觉得 Mario 跳得不够高，差一点点够不到上面的平台" |
| "在 AsciiElementRegistry 里注册一个新字符 W" | "我做了一个新机关，想在 ASCII 文本里用一个字符代表它" |

### 描述体验，不描述实现

| 不要这样说 | 要这样说 |
|-----------|----------|
| "给 FireTrap 加一个 OnTriggerEnter2D 的检测" | "火焰陷阱现在只烧上面的人，我想让旁边经过的人也能被烧到" |
| "在 Update 里每帧检测距离" | "我想让 Mario 靠近伪装的 Trickster 时有一个微弱的提示" |

### 验证结果，不验证过程

你不需要检查 AI 写的代码是否"优雅"。你只需要：
1. 跑一遍看效果对不对。
2. 跑一遍自动化测试看有没有破坏旧功能。
3. 如果都没问题，就是好的。
