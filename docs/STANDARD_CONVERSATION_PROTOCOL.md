# MarioTrickster 标准对话协议（Standard Conversation Protocol）

> **版本**：V1.0  
> **生效日期**：2026-04-12  
> **适用范围**：MarioTrickster 项目内所有“更新 / 升级 / 优化 / 续接 / 蒸馏 / 实现 / 验证 / 复盘”类新对话  
> **定位**：这是项目级协作宪章，不替代现有真相源；它负责把“如何开局、如何控场、何时停下、何时落库、如何收口”从习惯变成协议。

## 1. 制定本协议的原因

MarioTrickster 当前已经拥有 `SESSION_TRACKER.md`、`README.md`、`AI_WORKFLOW.md`、`docs/ART_PIPELINE_GUIDE.md`、`prompts/PROMPT_RECIPES.md` 等较强的仓库内记忆骨架，但近阶段暴露出的主要风险，已经不再是“完全没有流程”，而是**新对话过于随意、入口过宽、上下文恢复不稳定、形成结论后未及时落库、阶段边界不够硬**。这些问题一旦叠加，就会使项目进入“局部看似有产出、整体却逐步失控”的状态。

外部最佳实践与本仓库现状的交叉结论非常明确：**长项目不能依赖聊天气氛维持秩序，必须依赖固定入口、结构化任务模板、阶段性验证闸门与仓库内持久记忆**。因此，本协议的目标不是让提示词更“华丽”，而是让每次新对话都具备相同的起跑姿势、相同的风险控制骨架与相同的收口方式。

| 协议目标 | 解决的问题 | 预期收益 |
|---|---|---|
| **固定入口** | 新对话开局过于自由，容易漏读关键文档 | 降低续接失败率与误判当前阶段的概率 |
| **任务模板化** | 用户一句“帮我优化一下”会引发范围漂移 | 降低上下文污染，提升执行效率 |
| **验证闸门化** | AI 容易在范围不明或验证不足时继续推进 | 防止误改、过改和错误收口 |
| **回写沉淀化** | 可复用结论停留在聊天里，无法跨会话继承 | 提升项目长期记忆质量 |
| **异常显式化** | 模糊需求、证据不足、验证缺失时继续硬推 | 提高质量与可追溯性 |

## 2. 协议与现有仓库骨架的关系

本协议**不是另起一套平行体系**，而是对现有真相源进行编排增强。它只决定“何时读取、如何使用、何时必须回写”，并不替代这些文档本身的专业职责。

| 文档 | 继续承担的职责 | 本协议如何调用它 |
|---|---|---|
| `SESSION_TRACKER.md` | 新对话首读入口、当前状态、待办、回归、最小日常同步 | 作为**默认首读真相源** |
| `README.md` | 仓库首页导航、当前默认起步口径、入口索引 | 作为**入口索引与外部说明页** |
| `AI_WORKFLOW.md` | 用户侧提问、反馈、积分与 Git 流程说明 | 作为**人类侧操作说明书** |
| `docs/ART_PIPELINE_GUIDE.md` | 美术窄入口、验证顺序、量产前置条件 | 在美术相关任务中列为**必读专项文档** |
| `prompts/PROMPT_RECIPES.md` | 美术生产主配方库与后台硬规则 | 在蒸馏、LoRA、量产任务中列为**主规则库** |
| `docs/DISTILLATION_PROTOCOL.md` | 蒸馏、入库、验收与提交证据要求 | 在教程蒸馏与规则升级任务中列为**强制协议层** |
| `LevelStudio_DesignGuide.md` | 白盒关卡主线与关卡协作规则 | 在关卡、机制、布局任务中列为**主线专项入口** |

因此，**后续所有标准提示词都必须先服从本协议，再按任务类型加载专项文档**；不得反过来让某个单次任务把项目级纪律冲掉。

## 3. 不可省略的七条总则

> 这七条是所有标准提示词的硬骨架。用户可以追加个性化问题，但不能删除这七条要求。

### 3.1 先恢复上下文，再开始工作

任何新对话都必须先读取 `SESSION_TRACKER.md`，并按任务类型补读相关专项文档。**不得要求用户手动复述长上下文，不得把聊天历史当作长期记忆源。**

### 3.2 先审视现状与阶段，再提方案

复杂任务必须遵循：**现状判断 → 目标定义 → 范围边界 → 分阶段计划 → 执行 → 验证 → 落库 → 汇报**。不得在尚未确认项目当前阶段与主阻塞前，直接进入实现或改造。

### 3.3 先窄入口，后扩范围

MarioTrickster 当前阶段不是“每次都从零开始总规划”。默认优先使用**窄入口**：

| 任务主线 | 当前默认入口 |
|---|---|
| **关卡主线** | `先继续白盒关卡，不等最终美术。` |
| **美术支线** | `先做 trickster_style 本地验证，再决定切片量产。` |
| **蒸馏支线** | `先按蒸馏协议检查是否已有主规则，再决定新增。` |

只有当用户明确要求重开路线评估，才允许回到宽入口。

### 3.4 模糊即停，禁止脑补

需求边界、受影响文件、验收标准、用户目标不明确时，必须先澄清或给出带风险说明的方案分支。**不得把猜测当事实推进。**

### 3.5 形成结论必须落库

凡是能够提高后续质量、衔接效率、验证效率或减少返工的结论，包括规则、失败模式、参数甜区、命名约束、验收口径、默认入口、禁区和桥接结论，都不得只停留在聊天中。**未落库，不算真正完成。**

若本轮涉及 Git 写入，默认按 `master` 分支与 `https://github.com/jiaxuGOGOGO/MarioTrickster.git` 执行；提交后必须校验远程分支 commit SHA 与本地 `HEAD` 一致。若认证失效、推送失败或远程未同步，必须明确标记为 **“仅本地完成”**，不得把本地结果表述成已完成远程更新。

### 3.6 验证不足不得宣布完成

没有实际验证、没有列出未验证项、没有写清风险与下一步时，不得使用“已完成”叙述偷换概念。可以写“已完成文档改造”“已完成方案设计”，但不能冒充“生产闭环已完成”。

### 3.7 新协议必须兼容旧规则中的高价值部分

任何“精简提示词”“重写入口”“缩短模板”的行为，都必须遵守仓库已有的**防清洗原则**：只允许删除已被更强规则完整覆盖的内容，不允许把高价值重复、失败模式、结构保护逻辑、四区隔离、回写要求一起删掉。

## 4. 标准对话的固定执行顺序

每次使用标准提示词开启新对话时，默认执行顺序如下。

| 阶段 | 必须完成的动作 | 产出 |
|---|---|---|
| **A. 读档恢复** | 读取 `SESSION_TRACKER.md`，按任务类型补读专项文档 | 当前阶段、主阻塞、相关规则清单 |
| **B. 现状复述** | 用简洁语言复述“我理解的当前状态、目标、边界、风险” | 统一任务理解 |
| **C. 分阶段计划** | 列出计划阶段、拟修改范围、验证方式 | 可审查的行动骨架 |
| **D. 执行与记录** | 按计划实施修改、分析或研究 | 文件变更 / 方案 / 工单 / 结论 |
| **E. 验证与风险声明** | 写明已验证、未验证、仍需用户配合项 | 质量边界清晰 |
| **F. 落库与收口** | 将可复用结论写回真相源，并说明后续起步方式 | 跨会话稳定续接 |

这意味着，**标准提示词的核心不是“怎么发号施令”，而是“怎么锁死一个可靠的对话骨架”**。

## 5. 任务类型与必读文档映射

不同任务不应该加载完全相同的上下文。为了避免噪音膨胀，本协议采用“项目级常驻规则 + 任务型专项规则”的双层装载方式。

| 任务类型 | 必读文档 | 可选补读 | 默认禁止 |
|---|---|---|---|
| **新对话续接 / 综合升级** | `SESSION_TRACKER.md`、`README.md`、本协议 | `AI_WORKFLOW.md` | 直接跳过读档进入执行 |
| **关卡规划 / 白盒推进 / 机制微创** | `SESSION_TRACKER.md`、`LevelStudio_DesignGuide.md`、本协议 | `AUDIT_REPORT_S43.md` | 在未确认关卡目标前主动搞基建 |
| **美术验证 / 量产 / LoRA / 配方调用** | `SESSION_TRACKER.md`、`docs/ART_PIPELINE_GUIDE.md`、`prompts/PROMPT_RECIPES.md`、本协议 | `docs/WORKORDER_QA_STANDARD.md` | 直接从宽入口重新研究全套路线 |
| **教程蒸馏 / 规则入库 / 协议增强** | `SESSION_TRACKER.md`、`docs/DISTILLATION_PROTOCOL.md`、`prompts/PROMPT_RECIPES.md`、本协议 | PR 模板 | 只做总结、不做入库清单 |
| **Bug 修复 / 回归治理** | `SESSION_TRACKER.md`、`AI_WORKFLOW.md`、本协议 | 相关报告与测试文档 | 没有触发条件和验证项就直接改 |
| **交接文档 / 会话收口 / 存档** | `SESSION_TRACKER.md`、本协议 | `README.md` | 只说“已存档”却无回写证据 |

## 6. 四道验证闸门

为了防止“没看清楚就动手”“还没确认就扩散修改”“没验证就宣布完成”，本协议引入四道必须显式跨越的闸门。

### 6.1 闸门一：现状闸门

在开始任何改造前，必须明确以下四项：**当前阶段、主阻塞、当前默认入口、这次任务的目标边界**。如果其中任一项不清楚，优先补读或澄清。

### 6.2 闸门二：范围闸门

在编辑或重写前，必须说明：**计划动哪些文件、为什么动它们、哪些文件明确不动**。这样可以防止一次“优化提示词”意外演变成大面积文档漂移。

### 6.3 闸门三：验证闸门

任何方案、文档或实现都必须写明：**已验证什么、尚未验证什么、验证依赖谁、如果现在停止，项目处于什么状态**。没有这一步，就不允许使用“闭环”“完成”“可直接量产”等高确定性表述。

### 6.4 闸门四：落库闸门

可复用结论只有在被写回仓库后，才算真正完成。若因为权限、认证或用户确认等原因暂时无法写回，必须显式说明**待落库事项、目标文件与当前缺口**。

| 闸门 | 核心问题 | 未通过时应怎么做 |
|---|---|---|
| **现状闸门** | 我们现在到底处在哪个阶段？ | 先读档，不执行 |
| **范围闸门** | 这次到底改到哪里为止？ | 缩小范围或先确认 |
| **验证闸门** | 我怎么证明这次工作不是假完成？ | 列出验证项与未验证项 |
| **落库闸门** | 下次新对话凭什么还能接上？ | 回写真相源或列待回写清单 |

## 7. 标准汇报格式

所有标准提示词都应要求 AI 在关键节点按同一骨架汇报，避免“这轮说得很细、下轮说得很散”。推荐统一为以下五段式：

| 段落 | 内容要求 |
|---|---|
| **当前判断** | 当前阶段、主阻塞、本次任务目标 |
| **计划与范围** | 准备分几步做、预计修改哪些文件 |
| **执行结果** | 已做了什么、产出了什么 |
| **验证与风险** | 已验证、未验证、风险、边界 |
| **落库与下一步** | 已落到哪里、下一次建议怎么开局 |

这五段式本身就是一种**跨会话压缩格式**。它使后续接手者可以快速回答三个问题：做了什么、还差什么、下一步为什么是那个。

## 8. 用户自定义问题的合法插槽

用户完全可以在标准提示词里追加自己的偏好、疑问、限制和反馈，但必须放在**标准骨架之后**，而不是替代骨架。建议统一使用如下插槽：

```text
【我的补充要求】
- [你的个性化目标 / 偏好 / 特殊约束]
- [你希望重点关注的风险]
- [你希望我额外回答的问题]
```

这样做的意义是：**保留个性化，又不破坏稳定流程**。项目级纪律与本轮个性需求必须并存，而不是二选一。

## 9. 交接文档最小字段

无论是会话收口、handoff 备忘，还是阶段性总结，只要是面向“下次继续”的文档，都至少要能回答以下问题。

| 字段 | 必须说明什么 |
|---|---|
| **任务标题** | 这份交接对应的是什么任务 |
| **时间戳** | 文档写于何时 |
| **当前阶段** | 项目/支线目前在哪个阶段 |
| **已完成项** | 已完成到什么程度，产出了什么 |
| **未完成项** | 仍缺什么，不能假装闭环 |
| **主阻塞 / 风险** | 当前最大的不确定性是什么 |
| **下一步建议** | 下次最推荐从哪一步接续 |
| **证据位置** | 相关文档、文件、路径或报告在哪 |

这组字段与项目现有 `SESSION_TRACKER.md` 的功能相容，也吸收了外部 handoff 实践中“低技术但高兼容”的优点。

## 10. 当前阶段下的默认执行口径

结合 `SESSION_TRACKER.md` 当前状态，本协议把 MarioTrickster 在 2026-04-12 的**默认起步口径**固定如下：

| 主线 | 当前默认动作 | 为什么 |
|---|---|---|
| **A 轨：关卡白盒主线** | 继续白盒关卡，不等待最终美术 | 项目唯一上游任务源仍是关卡推进 |
| **B 轨：美术使能支线** | 先完成 `trickster_style` 本地验证 | 当前主阻塞是 LoRA 验证结果缺失，而不是缺新研究 |
| **C 轨：资产回接桥接层** | 仅在已有命名资产需求时推进接回定义与切片量产 | 防止美术支线重新膨胀成无上游约束的泛探索 |

因此，如果用户只是说“帮我优化项目”，标准提示词默认不应直接导向“大而全重构”，而应先把任务锚定到这三个轨道中的一个，再开始行动。

## 11. 何时允许偏离标准骨架

标准骨架并不是为了僵化，而是为了在以下情况出现前始终保持稳定。只有出现这些条件时，才允许偏离默认结构：

| 触发条件 | 允许的偏离 | 仍不可突破的底线 |
|---|---|---|
| **用户明确要求全局路线重估** | 可从窄入口回到总入口诊断 | 仍需先读档、后计划 |
| **任务极小且边界极清楚** | 可缩短计划描述 | 仍需说明范围与结果 |
| **紧急积分 / 时间压力** | 可优先做存档与最小回写 | 不得省略当前状态与待办 |
| **只读型研究任务** | 可不涉及代码或推送 | 可复用结论仍需落库或至少形成研究文档 |

---

## 12. 执行结论

MarioTrickster 后续所有“需要更新、升级、优化项目”的新对话，都不应再以随意自然语言直接开局，而应统一改为：**先套用标准提示词骨架，再把本轮需求填进去，再把个性化问题放入补充插槽**。这样做的价值，不在于形式整齐，而在于它同时保证了**项目衔接、质量、效率与失控风险控制**四个目标。

下一步的真正落地方式，不是让用户去记住本协议全文，而是提供一套**可直接复制使用的标准提示词模板库**。因此，本协议应与 `prompts/STANDARD_PROJECT_PROMPTS.md` 配套使用。
