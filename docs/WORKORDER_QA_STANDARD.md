# 📋 工单质量标准 (Work Order QA Standard)

> **本文档定义了 Manus TA 下发 ComfyUI 工单时必须遵守的格式与内容标准。**
> 旧版工单 S81 因节点缺失、参数名不匹配中文 UI、连线描述抽象等问题，导致用户无法直接操作。本标准从 S82 起强制执行。

---

## 一、工单必备四区

每份工单必须包含以下四个区域，缺一不可。

| 区域 | 必须包含 | 禁止出现 |
|---|---|---|
| **节点清单区** | 完整的节点列表（含中文 UI 搜索名 + 英文原名），标注哪些节点是用户可能遗漏的 | 仅写英文概念名而不给中文搜索名 |
| **参数区** | 每个节点的每个需要修改的参数，格式为「参数名：默认值 → 目标值」 | 仅写目标值不写默认值（用户无法确认自己改对了） |
| **连线区** | 端口对端口的连线表格，格式为「起点节点.端口名 → 终点节点.端口名」 | 用伪代码或箭头流程图描述连线 |
| **故障排除区** | 「现象 → 调哪个节点的哪个参数 → 调到多少」的三列表格 | 仅写"降低权重"不说降到多少 |

---

## 二、节点命名铁律

工单中提到的每个节点，必须同时给出以下两个名称：

| 格式 | 示例 |
|---|---|
| **中文 UI 显示名**（用户在画布双击搜索时看到的） | `Checkpoint加载器（简易）` |
| **英文原名**（括号内标注，方便搜索不到时用英文搜） | (CheckpointLoaderSimple) |

如果某个节点在中文 UI 中的名称与英文原名差异较大，必须额外标注搜索关键词。

---

## 三、参数标注格式

对于每个需要用户修改的参数，必须使用以下格式：

```
参数中文名 (英文名)：默认值 → 目标值
```

示例：
```
步数 (steps)：20 → 35
cfg：8.0 → 6.0
采样器名称 (sampler_name)：euler → euler_ancestral
```

对于不需要修改的参数，用一句话说明「保持默认即可」，不要逐一列出。

---

## 四、连线表格格式

连线必须使用四列表格，表头必须明确标注 **输出口** 和 **输入口** 的方向，禁止使用“起点端口/终点端口”这类不明确方向的表述：

| 从这个节点 | 的输出口▶（节点右侧） | 拖线到这个节点 | 的◀输入口（节点左侧） |
|---|---|---|---|
| Checkpoint加载器 | 输出口▶ **模型** | K采样器 | ◀输入口 **模型** |

这样即使两个端口名字相同（如都叫 `model`），用户也能通过“输出口▶”和“◀输入口”的标记分清方向。当一个输出口需要连接到多个目标时，分成多行写。

---

## 五、模型文件引用规则

工单中引用的所有模型文件，必须满足以下条件：

| 规则 | 说明 |
|---|---|
| **以用户实际有的模型为准** | 参照 `ART_PIPELINE_GUIDE.md` 步骤 0 中列出的已安装模型 |
| **明确标注放置路径** | 如 `models/checkpoints/` 或 `models/ipadapter/` |
| **备选模型单独标注** | 用户没有时给出下载链接，但不作为默认选项 |

## 六、算力节约原则

| 原则 | 具体要求 |
|---|---|
| **先跑 1 张验证再批量** | 工单必须标注"先用批次1的参数跑1张，确认画风OK后再批量" |
| **batch_size 保持 1** | 探索期禁止 batch_size > 1，避免一次性浪费多张算力 |
| **参数甜区优先** | CFG 必须在 4.0-7.0 甜区内，steps 必须在 25-40 范围内 |
| **禁止冗余节点** | 工单中的每个节点必须有明确用途，不允许"备用"或"可选"节点 |

## 七、防歧义铁律（源自 S81 V2 的教训）

工单中的节点描述必须避免以下五类歧义：

| 歧义类型 | 禁止做法 | 正确做法 |
|---|---|---|
| **分组歧义** | 把多个独立节点写在同一个标题下（如“节点名 A / 节点名 B”），让用户误以为它们是同一个节点的子功能 | 每个独立节点必须单独占一个编号标题，并在说明中注明“这是一个独立节点，需要单独拉出” |
| **下拉选项歧义** | 写一个用户实际 UI 中不存在的选项名（如写 `STANDARD (SDXL)` 但实际显示的是 `STANDARD (medium strength)`） | 必须使用用户 UI 中实际显示的完整文字；如果不确定用户版本的确切文字，必须标注“请确认你的下拉菜单中是否有这个选项” |
| **同类多实例歧义** | 同一种节点需要拉出多个时（如两个 CLIP文本编码），只写“节点名 A / B”而不解释 | 必须明确写“拉出第一个”“拉出第二个”，并说明为什么需要两个以及它们分别接哪里 |
| **口语化描述歧义** | 用口语描述操作而不对应具体参数（如写“保持随机”但该输入框只能填数字，“随机”实际由另一个参数控制） | 每条指令必须精确到“哪个参数、填什么值”；如果一个功能由多个参数配合实现，必须逐个列出并解释各自职责 |
| **端口方向歧义** | 连线表中只写“起点端口 / 终点端口”，当两个端口同名时（如都叫 `model`）用户无法分清是从哪边出来、接到哪边进去 | 表头必须用“输出口▶（节点右侧）”和“◀输入口（节点左侧）”明确标注方向，让用户知道从节点右边拖到另一个节点左边 |

## 八、自检清单（Manus TA 下发前必过）

下发工单前，Manus TA 必须逐项确认：

- [ ] 节点清单是否完整？（用户按清单拉完所有节点后，不会缺少任何连线目标）
- [ ] 每个节点的中文搜索名是否准确？（用户在画布双击搜索能直接找到）
- [ ] 参数是否标注了「默认值 → 目标值」？（用户能确认自己改对了）
- [ ] **下拉选项是否使用了用户 UI 中实际显示的文字？**（而不是官方文档或英文概念名）
- [ ] **每个独立节点是否单独占一个编号标题？**（禁止把多个独立节点写在同一个标题下）
- [ ] **是否存在口语化描述？**（每条指令必须能直接对应到 UI 上的某个具体参数和具体值，禁止“保持随机”“调高一点”等模糊表述）
- [ ] 连线表格是否覆盖了所有端口？（没有悬空的输入口）
- [ ] 引用的模型文件是否在用户已安装清单中？
- [ ] 故障排除是否给出了具体数值而非模糊描述？
- [ ] **（LoRA 验证工单）全局设置是否从用户实际工作流截图提取？**（禁止用通用推荐值）
- [ ] **（LoRA 验证工单）文件命名是否利用了 ComfyUI 自动编号？**（禁止要求用户逐张改名）
- [ ] **（LoRA 验证工单）是否附带顺序号→参数对照表？**（映射负担禁止转嫁给用户）
- [ ] **（防呆）是否明确说明了"不需要上传图片"或"需要上传什么图片"？**（禁止让用户猜）
- [ ] **（防呆）是否明确列出了"全程不能动"的参数清单？**（如负向词、Seed 等）
- [ ] **（防呆）引入新模型时是否提供了下载链接+存放路径+刷新方法？**（三件套缺一不可）
- [ ] **（防呆）修改用户偏好参数时是否解释了原因？**（禁止只给目标值不说为什么）
- [ ] **（防呆）批量跑图时是否给出了逐张操作节奏？**（禁止只说"循环执行"）
- [ ] **（防呆）文件命名是否利用了自动编号？**（禁止要求逐张手动改名）

## 九、LoRA 验证工单专项规则（源自 S96 的教训）

本节规则适用于所有“LoRA 本地验证”类型的工单。S96 期间暴露了三个问题：全局设置与用户实际环境不匹配、文件命名要求繁琐导致执行效率低下、参数映射负担转嫁给用户。以下规则从 S96 起强制执行。

### 9.1 全局设置必须从用户实际工作流提取

| 规则 | 说明 |
|---|---|
| **禁止用“推荐默认值”代替用户实际配置** | 工单中的 Steps、CFG、Sampler、Scheduler、分辨率、负面词等全局参数，必须从用户提供的工作流截图或历史记录中提取，不得自行填写通用推荐值 |
| **必须先问再写** | 如果用户未提供工作流截图，必须先要求用户提供，不得先下发工单再补问 |
| **差异项必须显式标注** | 如果工单中的某个参数与用户常用配置不同（例如验证要求必须用特定值），必须用「❗ 与你常用配置不同」标注并说明原因 |

### 9.2 文件命名必须利用 ComfyUI 自动编号

| 规则 | 说明 |
|---|---|
| **禁止要求用户逐张手动改名** | 用户每个题材只改一次文件名前缀（如 `S` / `A` / `C`），ComfyUI 自动递增编号 |
| **禁止把 Seed 写进文件名** | Seed 在回传模板中填写即可，不得要求用户手动拼接到文件名中 |
| **前缀必须简短** | 前缀仅用 1~2 个字母标识题材，禁止超过 5 个字符 |

### 9.3 参数映射由 AI 侧负责，禁止转嫁给用户

| 规则 | 说明 |
|---|---|
| **必须附带「顺序号 → 参数对照表」** | 工单必须包含一张映射表，明确每个自动编号对应的模式、权重和触发词位置 |
| **用户只需保证顺序** | 用户的唯一义务是“每个题材内严格按 1~10 顺序跑”，其余映射工作全部由 AI 在收图后完成 |
| **回传模板必须极简** | 回传模板只问两件事：① Seed 数字；② 是否严格按顺序跑。禁止要求用户填写任何其他内容 |
| **禁止要求用户填写主观判定** | “最佳权重”“表现简述”“是否污染”“去污词”等判定项全部由 AI 看图后完成，禁止转嫁给用户。用户不是专业美术人员，不应被要求做专业判断 |

	### 9.4 工作流改造必须基于用户现有工作流
	
	| 规则 | 说明 |
	|---|---|
	| **禁止默认要求用户从零搭建** | 必须先确认用户是否已有可用工作流，优先在现有工作流上改造（Bypass 不需要的节点、插入新节点） |
	| **改造步骤必须编号并明确** | 每步必须写清“哪个节点 → 什么操作 → 改成什么”，禁止笼统描述 |
	| **改完必须要求用户截图确认** | 改造完成后必须要求用户截图发回，由 AI 检查连线是否正确后才允许开跑 |

	## 十、工单指引防呆规则（源自 S97 动画工单的教训）
	
	本节规则适用于所有包含 ComfyUI 操作指引的工单。S97 期间暴露了工单指引不够保姆级、缺乏预判、操作节奏缺失等问题。以下规则从 S97 起强制执行。
	
	### 10.1 零假设原则（禁止假设用户已知操作）
	
	| 规则 | 说明 |
	|---|---|
	| **必须提供操作级指引** | 提到任何功能时，必须说明具体点哪里。例如：不能只说"切换到下一张图"，必须说"点击加载图像节点上的 ▶ 右箭头切换到下一张图" |
	| **必须说明纯文生图的特殊性** | 如果工作流不需要上传图片，必须在开头显式声明："本次为纯文生图，不需要上传任何图片" |
	| **必须说明哪些参数绝对不能动** | 除了要求修改的参数外，必须明确列出"全程保持不动"的参数（如负向提示词、Seed 等） |
	
	### 10.2 依赖预判与完整闭环
	
	| 规则 | 说明 |
	|---|---|
	| **必须预判模型缺失** | 引入新模型（如 ControlNet、特定 LoRA）时，必须假设用户本地没有，主动提供：① 确切的下载链接或文件名；② 准确的存放路径（如 `models/controlnet/`）；③ 刷新/重启 ComfyUI 的具体方法 |
	| **必须解释参数修改原因** | 要求用户修改偏好参数（如 CFG 从 8.0 改为 6.0，LoRA 权重设为 0.6）时，必须用一句话解释原因，否则用户可能会按自己的习惯改回去 |
	
	### 10.3 操作节奏与自动化
	
	| 规则 | 说明 |
	|---|---|
	| **必须给出明确的"操作节奏"** | 批量跑图时，必须给出逐张操作的循环说明。例如："第1张：改参数A → 出图 → 保存；第2张：只改参数B → 出图 → 保存" |
	| **必须提供最省力的命名方案** | 强制利用 ComfyUI 的 `保存图像` 节点前缀功能实现自动编号，禁止让用户逐张手动重命名 |
	
	### 10.4 用户环境快照必须持续更新
	
	| 规则 | 说明 |
	|---|---|
	| **每次用户发截图必须提取并回写** | 用户发来的工作流截图中包含的实际参数（Steps、CFG、Sampler、IPAdapter preset/weight/end_at、ControlNet 强度等）必须提取并回写到本文档第五条的模型清单或专项工单中 |
	| **新增模型必须同步更新清单** | 用户每次下载并安装新模型后，必须将其添加到第五条的“用户已安装模型清单”中 |
	| **工单参数必须以最新快照为准** | 下发工单时，必须以用户最后一次发来的截图中的实际参数为准，而不是历史工单中的参数 |
	
	### 10.5 回传流程极简化（通用规则）
	
	| 规则 | 说明 |
	|---|---|
	| **用户只需发图+填最少信息** | 回传模板最多只问：① Seed 数字；② 是否按顺序跑。其余判定全部由 AI 完成 |
	| **禁止要求用户做专业判断** | 用户不是美术专业人员，禁止要求填写“最佳权重”“表现简述”“污染判断”等专业内容 |
	| **图片可以直接发不用打包** | 明确告诉用户可以直接发图片，不需要打压缩包 |
