# 📋 工单质量标准 (Work Order QA Standard)

> **本文档定义了 Manus TA 下发 ComfyUI 工单时必须遵守的格式与内容标准。**
> 旧版工单 S81 因节点缺失、参数名不匹配中文 UI、连线描述抽象等问题，导致用户无法直接操作。本标准从 S82 起强制执行。

---

## 一、工单必备四区

每份工单必须包含以下四个区域，缺一不可。

| 区域 | 必须包含 | 禁止出现 |
|---|---|---|
| **节点清单区** | 完整的节点列表（含中文 UI 搜索名 + 英文原名），标注哪些节点是用户可能遗漏的 | 仅写英文概念名而不给中文搜索名 |
| **参数区** | 每个节点的每个需要修改的参数，格式为「参数名：默认值 → 目标值」 | 仅写目标值不写默认值（用户无法确认自己改对了） |
| **连线区** | 端口对端口的连线表格，格式为「起点节点.端口名 → 终点节点.端口名」 | 用伪代码或箭头流程图描述连线 |
| **故障排除区** | 「现象 → 调哪个节点的哪个参数 → 调到多少」的三列表格 | 仅写"降低权重"不说降到多少 |

---

## 二、节点命名铁律

工单中提到的每个节点，必须同时给出以下两个名称：

| 格式 | 示例 |
|---|---|
| **中文 UI 显示名**（用户在画布双击搜索时看到的） | `Checkpoint加载器（简易）` |
| **英文原名**（括号内标注，方便搜索不到时用英文搜） | (CheckpointLoaderSimple) |

如果某个节点在中文 UI 中的名称与英文原名差异较大，必须额外标注搜索关键词。

---

## 三、参数标注格式

对于每个需要用户修改的参数，必须使用以下格式：

```
参数中文名 (英文名)：默认值 → 目标值
```

示例：
```
步数 (steps)：20 → 35
cfg：8.0 → 6.0
采样器名称 (sampler_name)：euler → euler_ancestral
```

对于不需要修改的参数，用一句话说明「保持默认即可」，不要逐一列出。

---

## 四、连线表格格式

连线必须使用四列表格，表头必须明确标注 **输出口** 和 **输入口** 的方向，禁止使用“起点端口/终点端口”这类不明确方向的表述：

| 从这个节点 | 的输出口▶（节点右侧） | 拖线到这个节点 | 的◀输入口（节点左侧） |
|---|---|---|---|
| Checkpoint加载器 | 输出口▶ **模型** | K采样器 | ◀输入口 **模型** |

这样即使两个端口名字相同（如都叫 `model`），用户也能通过“输出口▶”和“◀输入口”的标记分清方向。当一个输出口需要连接到多个目标时，分成多行写。

## 五、模型文件引用规则

工单中引用的所有模型文件，必须满足以下条件：

| 规则 | 说明 |
|---|---|
| **以用户实际有的模型为准** | 参照 `ART_PIPELINE_GUIDE.md` 步骤 0 中列出的已安装模型 |
| **明确标注放置路径** | 如 `models/checkpoints/` 或 `models/ipadapter/` |
| **备选模型单独标注** | 用户没有时给出下载链接，但不作为默认选项 |

## 六、算力节约原则

| 原则 | 具体要求 |
|---|---|
| **先跑 1 张验证再批量** | 工单必须标注"先用批次1的参数跑1张，确认画风OK后再批量" |
| **batch_size 保持 1** | 探索期禁止 batch_size > 1，避免一次性浪费多张算力 |
| **参数甜区优先** | CFG 必须在 4.0-7.0 甜区内，steps 必须在 25-40 范围内 |
| **禁止冗余节点** | 工单中的每个节点必须有明确用途，不允许"备用"或"可选"节点 |

## 七、防歧义铁律（源自 S81 V2 的教训）

工单中的节点描述必须避免以下五类歧义：

| 歧义类型 | 禁止做法 | 正确做法 |
|---|---|---|
| **分组歧义** | 把多个独立节点写在同一个标题下（如“节点名 A / 节点名 B”），让用户误以为它们是同一个节点的子功能 | 每个独立节点必须单独占一个编号标题，并在说明中注明“这是一个独立节点，需要单独拉出” |
| **下拉选项歧义** | 写一个用户实际 UI 中不存在的选项名（如写 `STANDARD (SDXL)` 但实际显示的是 `STANDARD (medium strength)`） | 必须使用用户 UI 中实际显示的完整文字；如果不确定用户版本的确切文字，必须标注“请确认你的下拉菜单中是否有这个选项” |
| **同类多实例歧义** | 同一种节点需要拉出多个时（如两个 CLIP文本编码），只写“节点名 A / B”而不解释 | 必须明确写“拉出第一个”“拉出第二个”，并说明为什么需要两个以及它们分别接哪里 |
| **口语化描述歧义** | 用口语描述操作而不对应具体参数（如写“保持随机”但该输入框只能填数字，“随机”实际由另一个参数控制） | 每条指令必须精确到“哪个参数、填什么值”；如果一个功能由多个参数配合实现，必须逐个列出并解释各自职责 |
| **端口方向歧义** | 连线表中只写“起点端口 / 终点端口”，当两个端口同名时（如都叫 `model`）用户无法分清是从哪边出来、接到哪边进去 | 表头必须用“输出口▶（节点右侧）”和“◀输入口（节点左侧）”明确标注方向，让用户知道从节点右边拖到另一个节点左边 |

## 八、自检清单（Manus TA 下发前必过）

下发工单前，Manus TA 必须逐项确认：

- [ ] 节点清单是否完整？（用户按清单拉完所有节点后，不会缺少任何连线目标）
- [ ] 每个节点的中文搜索名是否准确？（用户在画布双击搜索能直接找到）
- [ ] 参数是否标注了「默认值 → 目标值」？（用户能确认自己改对了）
- [ ] **下拉选项是否使用了用户 UI 中实际显示的文字？**（而不是官方文档或英文概念名）
- [ ] **每个独立节点是否单独占一个编号标题？**（禁止把多个独立节点写在同一个标题下）
- [ ] **是否存在口语化描述？**（每条指令必须能直接对应到 UI 上的某个具体参数和具体值，禁止“保持随机”“调高一点”等模糊表述）
- [ ] 连线表格是否覆盖了所有端口？（没有悬空的输入口）
- [ ] 引用的模型文件是否在用户已安装清单中？
- [ ] 故障排除是否给出了具体数值而非模糊描述？
