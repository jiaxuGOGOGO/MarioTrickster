# 项目协作提示词深改研究笔记（2026-04-12）

## 已读取来源 1：GitHub Blog《How to build reliable AI workflows with agentic primitives and context engineering》

提炼到的高价值原则如下：

| 主题 | 可转译为 MarioTrickster 协作协议的结论 |
|---|---|
| **上下文工程** | 应把“读取哪些文件、什么时候读取、为什么读取”做成固定上下文注入点，而不是依赖自由发挥。 |
| **结构化思考** | 复杂任务应强制分阶段：审视现状、计划、执行、验证、沉淀、汇报。 |
| **验证闸门** | 在关键节点加入“停止并等待用户确认”或“未验证不得进入下一阶段”的门槛。 |
| **会话切分** | 不同任务阶段应使用不同的固定入口话术，降低上下文污染。 |
| **模块化规则** | 只加载当前任务相关规则，避免所有规范同时压入导致噪音上升。 |
| **记忆驱动开发** | 项目知识与关键决策必须固化为仓库内文档，聊天记录不可作为长期记忆源。 |
| **工作流模板化** | 标准提示词不应只是“开场白”，而应是一套可重复执行的工作流模板。 |
| **计划—任务—实现链条** | 提示词最好显式要求先计划，再拆任务，再执行并验证。 |

## 已读取来源 2：GitHub awesome-copilot《AI Prompt Engineering & Safety Best Practices》

提炼到的高价值原则如下：

|---|---|
| **明确任务** | 提示词必须写清任务目标、输出结构、约束与边界，不能只说“帮我优化一下”。 |

| **自动化评估维度** | 可用准确性、相关性、一致性、效率与安全性作为提示词/协作效果评价维度。 |
| **版本管理** | 提示词与协作协议要有版本号、变更原因与迁移说明。 |

## 与 MarioTrickster 当前现状的直接对照

| 项目现状 | 外部最佳实践的启发 |
|---|---|
| 已有 `SESSION_TRACKER.md`、`README.md`、`ART_PIPELINE_GUIDE.md` 等真相源 | 说明项目已经具备“上下文工程”骨架，重点不是重建，而是把入口、闸门、验收与异常处理标准化。 |
| 用户担心新对话过于随意导致项目失控 | 需要新增“标准开场提示词 + 任务类型模板 + 禁止随意开局条款 + 异常上报规则”。 |
| 当前已明确本地 commit 与远端 push 必须分开汇报 | 这正属于“验证闸门”与“状态汇报标准化”，应升格为正式协议。 |
| 当前任务入口已固定为窄入口 | 应把不同任务类型映射成不同标准提示词模板，而不是继续依赖宽泛自由输入。 |
| 用户希望长期维护项目质量与衔接效率 | 需要形成“项目协作宪章 + 标准提示词模板库 + 用户反馈补充槽位”的组合文档。 |

## 已读取来源 3：GitHub 仓库《cowork-context-framework》

提炼到的高价值原则如下：

| 主题 | 可转译为 MarioTrickster 协作协议的结论 |
|---|---|
| **入口恢复机制** | 每次新会话开头都应读取“入口文件 + 当前项目状态 + 最新会话记录”，不能依赖聊天历史。 |
| **稳定流程与环境细节分离** | 核心协作协议应短而稳定，工具细节、环境差异、升级路径应拆到独立文件。 |
| **自动累积文档** | 需求、设计、验证、状态变更、会话结果要分别回写到对应文档，而不是混在单一日志。 |
| **渐进式补全** | 不只是记录结果，还要在阶段门审查时主动补齐缺失信息。 |
| **意图/里程碑/任务/ADR 可追溯** | 项目应能回答“为什么做、做到哪、由谁决策、证据是什么”。 |
| **升级兼容** | 新协议落地时应区分新增、替换、合并、跳过，不能粗暴覆盖项目特有内容。 |

## 当前形成的设计判断

综合四个来源，MarioTrickster 的“标准提示词深改”不应只是加一段更强硬的开场白，而应形成四层结构：

1. **固定入口层**：新会话强制读取哪些真相源、先做什么检查；
2. **任务模板层**：按续接、规划、实现、调优、复盘等类型给出标准提示词；
3. **验证闸门层**：每一阶段什么情况下必须停下汇报、什么情况下禁止继续；
4. **回写沉淀层**：本次会话结束后必须把哪些结论写回仓库。

这意味着后续改造重点应放在“协议化”和“模板化”，而不是单纯润色措辞。

| **Keep instruction files concise** | 核心提示词要短、稳、常驻；细则拆到专项文档，避免每轮对话上下文膨胀。 |
| **Plan first, then implement** | 标准提示词必须默认要求先做计划，再做实现。 |
| **Provide expected output for verification** | 每次任务都应给出验收标准、预期结果或测试要求，便于自校验。 |
| **Ask clarifying questions when ambiguous** | 任务描述模糊时应先澄清，不得自行脑补。 |
| **Manage context and sessions** | 长项目必须管理会话边界与交接，而不是依赖连续聊天。 |

## 研究阶段中间结论

到目前为止，外部资料与 MarioTrickster 现状已经高度收敛到同一方向：

1. **标准提示词必须工程化**，不是礼貌措辞优化；
2. **项目级常驻规则必须短而稳定**，专项细则分流到文档；
3. **新对话必须先恢复上下文，再动手**；
4. **所有复杂任务必须“先计划、再确认、再执行、再验证、再落库”**；
5. **用户自定义问题应被允许，但只能放在标准骨架之后补充**，不能冲掉骨架。
